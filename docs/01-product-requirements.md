# 产品需求说明

文档状态：设计基线 0.5

目标读者：产品设计、客户端开发、服务端开发、Agent/工具开发、测试

相关文档：[领域模型](02-domain-model.md) · [权限模型](03-permission-model.md) · [系统架构](04-system-architecture.md)

## 1. 产品定义

NGAPD 是面向独立游戏开发团队的 AI 原生敏捷项目管理系统。系统使用递归任务表达从项目目标到具体执行工作的完整分解过程，并提供用户级、项目级和任务级三种独立工作区，使人和 Agent 在稳定、分层、可追溯的上下文中协作。

产品不是传统 Jira 的完整替代品，也不是通用网盘或源代码管理系统。它重点解决以下问题：

- 小团队成员往往承担多个专业角色，固定职位无法准确表达能力范围。
- 传统任务系统只保存状态和最终结果，AI 协作产生的大量中间分析无法沉淀。
- Agent 缺少稳定的任务边界，容易混入无关上下文或修改越权对象。
- 项目知识分散在聊天、临时文档和个人目录中，无法随项目演进形成知识库。

## 2. 产品目标

### 2.1 核心目标

- 提供结构简单、可无限递归分解的任务模型。
- 以同级 DAG 表达局部任务执行顺序。
- 以逻辑角色描述成员能力，而非把能力与权限绑定。
- 让用户、项目和任务分别拥有独立的本地/服务端工作区，并形成个人偏好、项目规范和任务过程三个上下文层级。
- 允许 Agent 在权限和人工确认约束下操作项目与任务。
- 将任务过程文档、决策和完成摘要组织为可追溯 Wiki。
- 在少于 20 人的团队中保持低管理成本。

### 2.2 非目标

第一阶段不以以下能力为目标：

- 企业级组织架构、部门、复杂 RBAC/ABAC 策略编辑器。
- 可任意定制的工作流和字段系统。
- 源代码托管、分支管理或 Git 平台替代。
- 通用云盘、大型二进制资产版本控制。
- 工时结算、财务、采购或外包合同管理。
- 多个连接同时写入同一个工作区。
- 无人监督、可绕过确认的自主 Agent。
- 服务端直接调用外部 AI、LLM 或其他云端内容处理 API。

## 3. 关键术语

| 术语 | 定义 |
|---|---|
| Project Owner | 项目的所有者，唯一；由 Project 上的单一字段指向一名活动成员，负责项目级配置和管理员任命 |
| Project Admin | `permission_level=admin` 的成员；只有主动开启管理员模式后才使用扩大权限 |
| 显式 Task Owner | Task 自身保存的可空 Owner |
| 有效 Task Owner | Task 自身显式 Owner 非空时取该 Owner；否则取最近一个显式 Owner 非空祖先的 Owner；一个 Task 始终最多只有一个有效 Owner |
| 逻辑角色 | 描述成员能力、等级和边界的 Agent 可读文本，不授予系统权限 |
| 任务正文 | 合并任务目标、工作描述和验收信息的单一富文本或 Markdown 字段 |
| 任务管理数据 | 标题、任务正文、截止时间、逻辑角色、标签、展示类型、父子关系、依赖、Owner、状态和归档状态等服务端数据 |
| 用户级工作区 | 每个用户一个；对应用户可写，其他已认证用户只读，用于个人流程、自用规则和 Skill |
| 项目级工作区 | 每个项目一个；Project Owner 和 Project Admin 可写，其他项目成员只读，用于项目通用规则、资料和 Skill |
| 任务级工作区 | 每个任务一个；有效 Task Owner 可写，其他项目成员只读，用于任务过程文档和交付物 |
| 逻辑工作区 | 用户、项目或任务创建时在服务端产生的工作区记录与初始内容 |
| 本地工作区 | 具有访问权限的用户首次连接逻辑工作区后，在本地物化的文件目录 |
| 写入租约 | 保证一个工作区同一时间只有一个有效写入会话的机制 |
| 有效状态 | 基础状态结合依赖阻塞、人工阻塞后呈现给用户的状态 |

## 4. 用户类型

### 4.1 项目所有者

- 创建和归档项目。
- 审批成员加入申请。
- 任命或取消 Project Admin。
- 开启管理员模式处理跨 Owner 的任务结构。
- 管理项目级规范和默认逻辑角色。
- 读取并写入项目级工作区。

### 4.2 项目管理员

- 默认仍以普通成员权限工作。
- 显式开启管理员模式后，可以处理任意任务的依赖、Owner、移动，以及顶层归档或非顶层不可恢复删除。
- 即使开启管理员模式，也不能写入其他 Task Owner 的任务级工作区。
- 人通过 Web 或本地客户端写入项目级工作区不要求开启管理员模式；Agent 写入项目级工作区时，必须由用户明确要求并确认进入管理员模式。

### 4.3 项目成员

- 维护自己拥有的任务及其任务级工作区。
- 创建、分解、推进和转交任务。
- 读取项目内任务和工作区知识。
- 读取项目级工作区和其他任务工作区；完整操作自己的用户级工作区与自己拥有的任务级工作区。
- 评论任务并接收必要通知。

## 5. 核心用户流程

### 5.1 创建项目

1. 用户创建项目并指定全局唯一的 Project Key。
2. 系统复制一份系统预设逻辑角色到项目中。
3. 系统创建项目级工作区、项目级上下文和根任务容器。
4. 创建者成为 Project Owner。
5. Project Owner 可以邀请成员或审批加入申请。

### 5.2 创建和分解任务

1. 成员在顶层或自己有权管理的任务下创建任务。
2. 顶层任务创建时必须指定一名活动成员作为显式 Owner；子任务可以指定任意活动成员，也可以把显式 Owner 留空并继承最近祖先的 Owner。Owner 指派立即生效，不需要接收者接受。
3. 系统事务性分配不可复用的任务编号，例如 `ABC-123`。
4. 服务端创建任务级逻辑工作区和初始任务文档。
5. 有效 Task Owner 可以继续创建子任务、填写统一任务正文并设置 UTC 截止时间；任一任务的有效 Owner 可以为其后代设置显式 Owner，因此顶层任务 Owner 对整个子树拥有 Owner 分配权。
6. 同级任务之间可以按照权限建立无环依赖。

### 5.3 Agent 连接任务

1. 用户要求 Agent 连接 `ABC-123`。
2. 系统校验用户身份、项目成员资格和任务读取权限。
3. 如果需要写入任务级工作区，继续校验用户是否为有效 Task Owner，并获取该工作区的独占写入租约。
4. 本地客户端物化或更新任务目录。
5. 系统把当前用户的用户级工作区规则/Skill、项目级工作区规则/Skill、当前用户在项目中的自我介绍和全部逻辑角色描述、当前任务完整信息、活动祖先当前信息与历史摘要、前置任务摘要和任务级工作区文件提供给 Agent。
6. Agent 在该上下文范围内执行工作，并记录工具操作。

### 5.4 完成任务

1. 有效 Task Owner 或其代表的 Agent 发起完成请求。
2. Agent 发起完成时必须获得人工确认。
3. 系统校验全部启用态直接子任务完成、依赖满足、无活动阻塞；条件首次满足时向有效 Task Owner 发送“可完成”提示。
4. 若存在有效写入租约，客户端先同步任务级工作区；否则服务端使用最新同步版本。服务端据此生成不可变完成快照。
5. 任务进入已完成状态，任务正文、展示字段、Owner、父子关系、依赖、阻塞和任务级工作区一并冻结；之后只允许添加评论、显式重新打开，以及对顶层任务执行归档。
6. 如果通过 Agent 完成任务，用户自己的 Agent 在完成提案中提交摘要草稿；服务端不调用任何外部模型。人工完成但没有摘要时，界面先提示“未生成摘要”，用户仍继续则由服务端根据任务名称、正文摘要、状态和完成时间生成确定性的一句话基本摘要。
7. 任务重新打开时创建新的工作周期，保留历史完成快照和摘要。

## 6. 功能需求

### 6.1 账号与项目

- `PRJ-001` 用户可以注册、登录和维护个人资料；注册成功时自动创建唯一用户级逻辑工作区。
- `PRJ-002` 用户可以配置默认逻辑角色和默认自我介绍。
- `PRJ-003` 用户可以创建、打开、归档和恢复项目；创建项目时自动创建唯一项目级逻辑工作区。产品不提供项目永久删除。
- `PRJ-004` Project Key 为 2—6 个大写英文字母，全局唯一且创建后不可修改。
- `PRJ-005` MVP 提供由 Project Owner 发起的项目级一致备份与恢复；备份必须覆盖项目业务数据、工作区文件与版本、对象内容、完成快照、知识条目和必要审计引用。
- `PRJ-006` 项目支持成员申请加入、审批、拒绝和移除。
- `PRJ-007` 项目支持中文项目说明和 Agent 可读项目规范；数据模型预留多语言能力。
- `PRJ-008` MVP 全部业务数据自托管，服务端不调用外部 API、AI 或 LLM；初期部署面向少于 10 名用户的小规模项目，通过内网或 VPN 访问一台 Linux Docker 服务器。

### 6.2 逻辑角色

- `ROL-001` 系统内置不可编辑的原始角色模板，首批模板见[逻辑角色模板](11-logical-role-templates.json)。
- `ROL-002` 创建项目时复制模板快照；系统模板后续变化不自动覆盖项目角色。
- `ROL-003` 项目内可以创建、编辑、归档和复制逻辑角色。
- `ROL-004` 角色包含名称、等级、能力、责任边界、禁区、任务类型和提示文本。
- `ROL-005` 一个项目成员可以绑定多个逻辑角色。
- `ROL-006` 逻辑角色不得直接授予 Web、工作区或 Agent 权限。

首批角色分为策划、技术、美术、音乐与声音四个角色族，其中技术包含程序开发与 QA。等级暂用 L1/L2/L3，分别表示指导下执行、独立负责专业模块、负责跨模块标准与最终决策；角色可以能力重叠，也不要求每个 Title 同时具有三个等级。

### 6.3 成员

- `MEM-001` Project Owner 可以任命 Project Admin。
- `MEM-002` Project Admin 只有开启管理员模式后才能使用管理权限。
- `MEM-003` 项目成员拥有项目内独立的角色绑定和自我介绍。
- `MEM-004` 新成员加入时从个人默认信息填充项目资料，之后互不自动同步。
- `MEM-005` 每个任务最多保存一个显式 Owner；显式 Owner 为空时继承最近一个显式 Owner 非空祖先的 Owner。顶层任务必须有显式 Owner，启用态任务开始执行前必须能够解析出有效 Owner。
- `MEM-006` 移除项目成员时，未完成任务中指向该成员的显式 Owner 统一置空并重新按祖先计算有效 Owner；无法解析有效 Owner 的任务不能进入进行中或已完成。已完成任务保持冻结并保留历史 Owner 引用，重新打开前必须由有权者指定活动 Owner。
- `MEM-007` Task Owner 指派不需要接收者接受。任一任务的有效 Owner 可以为其后代设置或替换显式 Owner；显式 Owner 可以把自己任务的显式 Owner 清空，使所有权回落到最近的显式 Owner 非空祖先。顶层任务的显式 Owner 不能清空。
- `MEM-008` Project Owner 被移除或停用前必须先把项目所有权转移给一名已接受转移的活动成员，项目不得处于无 Project Owner 状态。
- `MEM-009` Project Membership 的 `permission_level` 只包含 `admin / member`；Project Owner 由 Project 的唯一 `owner_membership_id` 表达，不在 `permission_level` 中重复保存；产品不设置 Viewer 角色。

### 6.4 任务树与依赖

- `TSK-001` 任务可以递归包含子任务，存储层不设置业务深度硬限制。
- `TSK-002` 任务编号在项目内单调分配，归档或删除后不复用。
- `TSK-003` 一个任务只能有一个父任务，且不能移动到自身子树。
- `TSK-004` 依赖只能连接同一父任务的直接子任务；顶层任务统一视为虚拟项目根节点的直接子任务，因此顶层任务之间也可以建立依赖。
- `TSK-005` 同一父任务下的依赖图必须无环。
- `TSK-005A` 不开启管理员模式时，当前用户拥有两个端点或两者共同的直接父任务，可以直接修改依赖；只拥有一个端点时可以发起依赖变更请求，但必须由另一端点 Owner 接受后才生效。
- `TSK-005B` 顶层任务的共同直接父级是虚拟项目根节点；Project Owner 视为该根作用域的控制者，可以在不开启管理员模式时修改顶层任务之间的依赖。
- `TSK-005C` 每个父级作用域维护独立的 `graph_version`；依赖提案、确认和执行必须绑定同一图版本，图发生变化后旧确认失效。
- `TSK-005D` 已完成任务的依赖关系冻结；不能新增、移除或改变任何以已完成任务为端点的依赖，必须先按重新打开策略处理相关已完成任务。
- `TSK-006` 任务完成的必要条件是全部处于启用态的直接子任务已完成；已删除或随归档顶层任务退出活动范围的节点不参与完成条件。
- `TSK-007` 子任务完成后父任务不自动完成，必须显式完成。
- `TSK-007A` 父任务满足全部完成条件时，系统将其标记为“可完成”并通知 Task Owner，但不自动改变任务状态。
- `TSK-008` 任务支持未开始、进行中、已阻塞、已完成四种有效状态。
- `TSK-009` 任务只使用 UTC 截止时间，不在 MVP 中维护预计工时；前端按设备系统时区显示。
- `TSK-010` 只有顶层任务可以归档；非顶层任务只能不可恢复地删除。归档或删除前必须计算对子树、他人任务和依赖的影响；删除还必须由操作者输入待删子树根任务的完整 Task Key，完全匹配后才可执行。仅为整理过多任务时应优先完成并归档顶层任务。
- `TSK-011` 完成任务下新增子任务前必须重新打开该任务。
- `TSK-012` 任务可以标记为普通、冲刺或里程碑；该标记只改变渲染外观，不参与状态、依赖、完成和权限计算。
- `TSK-013` 任务可以绑定一个可选的项目逻辑角色，表示执行该任务所需或推荐的主要能力；该字段不授予权限。
- `TSK-014` 顶层任务归档后，其依赖边保留为历史结构，但视为未启用，不参与阻塞、完成、环检测或排期计算。
- `TSK-015` 任务可以单向关注同项目的另一任务。关注只影响上下文发现和知识读取，不授予写权限，不参与状态、依赖、完成或排期计算；Agent 连接关注方任务后无需额外确认即可渐进读取被关注任务中当前用户原本有权读取的数据。
- `TSK-016` 任务完成时若显式 Owner 为空，系统先把当前有效 Owner 固化为显式 Owner；完成后除评论、显式重新打开和顶层归档外，任务字段、Owner、结构、阻塞、依赖和任务级工作区全部冻结。
- `TSK-017` 项目配置 `completed_successor_reopen_policy` 决定重新打开一个存在已完成后继的任务时如何处理：`deny` 拒绝重新打开；`cascade` 将依赖图中受影响的全部已完成 successor 闭包一并重新打开。默认值为 `deny`；跨 Owner 级联必须使用管理员模式并确认完整影响集合。

### 6.5 初版任务管理界面

- `VIS-001` 初版采用平铺树状导航，不实现语义缩放或嵌套画布。
- `VIS-002` 页面显示当前层级的任务列表，并显示同级任务之间的依赖关系。
- `VIS-003` 页面显示当前选中任务的标题、正文、显式/有效 Owner 及继承来源、逻辑角色、状态、本地化截止时间、展示类型和子任务统计。
- `VIS-004` 页面提供任务层级面包屑和“返回上一级”按钮。
- `VIS-005` 页面单独显示当前任务的直接子任务列表，用户可选择子任务进入下一层。
- `VIS-006` 支持按有效 Owner、任务绑定的逻辑角色、状态、截止时间和标签过滤。
- `VIS-007` 普通、冲刺、里程碑任务使用不同的图标、边框或色彩呈现，但交互和业务规则相同。
- `VIS-008` 支持按 Task Key 或标题搜索并定位到对应层级与任务。
- `VIS-009` 同层 200 个任务时列表、筛选、依赖展示和层级切换仍应保持可用。
- `VIS-010` 语义缩放、节点内嵌子图和无限画布作为后续版本能力，不属于初版验收范围。

### 6.6 工作区

- `WSP-001` 每个用户拥有且仅拥有一个用户级工作区，在用户注册时创建。
- `WSP-002` 每个项目拥有且仅拥有一个项目级工作区，在项目创建时创建。
- `WSP-003` 每个任务拥有且仅拥有一个任务级工作区，在任务创建时创建。
- `WSP-004` 用户级工作区只有对应用户可写，其他已认证用户只读。
- `WSP-005` 项目级工作区只有 Project Owner 和 Project Admin 可写，其他项目成员只读；人通过 Web 或本地客户端写入不要求开启管理员模式，Agent 写入则必须由用户明确要求并确认进入管理员模式。
- `WSP-006` 任务级工作区只有当前有效 Task Owner 可写，其他项目成员只读。
- `WSP-007` 三种工作区均允许多个只读连接，但同一工作区同一时间只能存在一个有效写入租约和一个可写连接。
- `WSP-008` 用户可在用户级工作区保存个人工作流程、自用规则和自用 Skill；可在项目级工作区保存项目通用规则、通用资料和通用 Skill。
- `WSP-009` 三种工作区都以文本为主，支持少量图片和可配置大小限制；服务端保存版本、校验和与快照。
- `WSP-010` 任务层级移动不得改变任务级工作区稳定路径。
- `WSP-011` 显式 Task Owner 变化或清空前必须处理任务级工作区租约：正常情况先同步并释放；持有者不可用时，有权者可以在影响预览后强制撤销，并以服务端最后同步版本继续。
- `WSP-012` 已完成任务的任务级工作区默认冻结；项目级和用户级工作区不随单个任务完成而冻结。
- `WSP-013` `.ngapd/` 下的控制元数据只由系统或本地客户端生成，`TASK.md` 和 `SUMMARY.md` 是服务端只读投影；三者都不参与用户内容同步，也不得由 Agent 写入。
- `WSP-014` 工作区服务端生命周期、本地副本状态和连接/租约状态分别建模，不把物化、只读连接和可写连接作为互斥的 Workspace 全局状态。
- `WSP-015` 租约失效或服务端版本不匹配后停止自动上传并进入冲突状态，不得静默覆盖。仍具有写资格的用户重新取得唯一租约后，必须明确选择“本地版本”或“服务端版本”作为新的唯一事实；选择本地会把完整本地清单提交为一个新服务端版本，选择服务端会覆盖本地受管文件。选择、差异摘要和结果必须审计。
- `WSP-016` MVP 提供项目级一致备份与原项目恢复，不提供任务回收站或单任务恢复。

### 6.7 Agent

- `AGT-001` Agent 工具与 Web UI 使用相同的服务端领域规则。
- `AGT-002` Agent 操作必须以当前用户身份执行，不拥有独立的越权身份。
- `AGT-003` Agent 必须同时通过 Web 权限、工作区权限和确认策略。
- `AGT-004` Agent 可以读取当前用户有权查看的全部任务管理信息，读取不需要人工确认。
- `AGT-004A` 底层权限允许读取其他用户的用户级工作区，但 Agent 不得自行发现或读取；只有用户明确指定目标用户、文件或目的后才能执行该读取。
- `AGT-005` Agent 对当前用户有写权限且已取得租约的用户级或任务级工作区具有完整访问权限，可以创建、修改、移动、重命名和删除其中的用户内容文件。
- `AGT-005A` Agent 写入项目级工作区除了要求当前用户是 Project Owner/Admin 并持有租约外，还必须由用户明确要求并确认进入管理员模式；进入后可以完整操作项目级用户内容文件。
- `AGT-006` Agent 对任务管理数据的任何修改都必须先形成操作提案并获得人工确认，包括修改标题、任务正文、UTC 截止时间或逻辑角色、创建任务、评论、状态、依赖、Owner、层级、展示类型、顶层归档、非顶层删除和完成。
- `AGT-007` Agent 必须显式进入管理员模式才能使用管理员权限；进入动作只能由用户确认触发，Agent 不能自动进入。
- `AGT-008` Agent 可以主动退出管理员模式，也应在任务完成、会话结束或不再需要管理权限时自动退出。
- `AGT-009` 所有 Agent 写操作必须记录输入、操作者、确认结果和关联任务；工作区文件操作同样保留本地/同步审计。
- `AGT-010` Agent 不得写入 `.ngapd/` 控制元数据、`TASK.md` 或 `SUMMARY.md`；服务端投影文件不能用于绕过任务管理确认。
- `AGT-011` Agent 提出完成任务时，应使用用户自己的模型根据完成快照来源生成摘要草稿并随完成提案提交；NGAPD 服务端只验证、保存和投影该摘要，不调用模型。

### 6.8 评论、通知与知识库

- `KNW-001` 项目成员可以对任务发表评论；评论不改变任务和工作区权限。
- `KNW-002` 支持 Owner 变更、被指派、阻塞、依赖变化、评论和本地化截止时间提醒。
- `KNW-002A` 父任务达到“可完成”条件时通知其 Task Owner，并对同一完成条件版本去重。
- `KNW-003` 用户可以配置非关键通知偏好，权限和破坏性操作通知不可关闭。
- `KNW-004` 完成任务后创建带来源链接的独立摘要条目；Agent 完成时优先保存用户自己的 Agent 生成并经用户确认的摘要，人工完成且未提供摘要时先提示，用户仍继续则保存服务端根据基本任务字段生成的确定性一句话摘要。摘要作为版本化 KnowledgeEntry 管理，不属于不可变完成快照中的可编辑文件。
- `KNW-005` Wiki 按任务树、任务绑定的逻辑角色、标签和全文搜索组织知识。
- `KNW-006` 重新完成任务时保留旧摘要和旧工作区快照；Owner 通过摘要接口修订 Agent 摘要或系统基本摘要，系统生成的 `SUMMARY.md` 只是只读投影。
- `KNW-007` Wiki 页面不能脱离任务来源静默覆盖历史内容。
- `KNW-008` 已完成任务仍允许活动项目成员添加评论；这些评论发布后不可编辑或删除，只能由管理员隐藏并保留审计副本。评论不改变冻结任务、完成快照或摘要来源。

## 7. 非功能需求

### 7.1 平台与部署

- 本地客户端：macOS、Windows。
- 服务端：Linux，Docker Compose 单机自托管；初期通过内网或 VPN 访问。
- Agent 协议和摘要模型应保持供应商无关。
- NGAPD 服务端不连接外部 AI、LLM 或其他云端内容处理 API；用户自己的 Agent 可以使用用户自行配置的模型。

### 7.2 目标规模

- 初期实际部署用户少于 10 人，项目规模较小，优先保证交付和单机可维护性。
- 典型项目成员少于 20 人。
- 典型同层任务少于 200 个，不作为业务硬限制。
- 工作区以文本为主，支持少量图片。
- 系统应使用分页、懒加载和当前层依赖渲染，不能依赖上述规模作为正确性前提。

### 7.3 一致性与审计

- 任务更新使用版本号进行乐观并发控制。
- 同级依赖图使用独立 `graph_version`；确认令牌必须绑定任务版本、图版本和影响集合摘要。
- 结构修改和 Agent 操作必须可审计。
- 写入租约、显式 Owner 变化和有效 Owner 重算必须原子化。
- 关键写操作提供幂等键，避免重试产生重复任务或依赖。

### 7.4 安全

- 本地客户端只能访问用户明确配置的 NGAPD 工作区根目录及其中已登记的用户级、项目级和任务级目录。
- Agent 工具不能通过路径穿越访问工作区外文件。
- Agent 工具不得写入 `.ngapd/` 控制元数据；读取其他用户的用户级工作区必须具有明确用户指令。
- 服务端对象存储路径不得直接使用用户输入。
- 工作区文件视为不可信内容，加载给 Agent 时必须与系统指令分隔。

## 8. MVP 范围

MVP 必须形成完整闭环，而不是只实现通用任务 CRUD：

1. 账号、项目、成员申请和简单权限。
2. 系统角色模板、项目角色和成员多角色绑定。
3. 递归任务、继承式单一有效 Owner、UTC 截止时间、任务逻辑角色、状态、任务关注和带图版本的同级 DAG。
4. 平铺树状任务界面：本级任务列表、当前任务详情、层级面包屑、直接子任务列表和返回上一级按钮。
5. 用户级、项目级、任务级逻辑工作区，本地物化、每工作区单写租约、冲突时由合格写入者选择唯一事实，以及文本/图片同步。
6. Agent 连接任务、读取全部可见任务信息和分层工作区上下文，完整操作当前用户有写权限且持有租约的用户级或任务级工作区；项目级写入还要求用户明确进入管理员模式；任何任务管理数据修改均通过人工确认。
7. 完成快照、由用户自己的 Agent 提交或由服务端基本字段兜底的摘要、基础 Wiki 浏览和搜索。
8. 评论、被指派通知、项目级一致备份与恢复和关键审计记录。

MVP 暂缓：语义缩放、嵌套任务图和无限画布、复杂通知规则、Git 平台深度集成、多人工作区编辑、可定制工作流、语义向量检索和移动客户端。

## 9. 产品级验收标准

当一个由策划、程序和美术组成的小团队能够完成以下流程时，MVP 才算成立：

1. 建立项目、角色和成员。
2. 把一个游戏功能拆成多层任务，并建立无环同级依赖。
3. 用户可以在用户级工作区保存个人流程，在项目级工作区保存团队规则，并在任务级工作区保存过程文档。
4. Project Admin 可以写项目级工作区但不能写他人的任务级工作区；其他项目成员只读项目级工作区。
5. 三种工作区在出现第二个写入连接时都拒绝授予新租约或要求显式接管。
6. Agent 在当前有写权限并取得租约的用户级或任务级工作区中完整操作用户内容；项目级写入还要求用户明确请求并确认管理员模式；任何作用域都不能写保护路径或无权限工作区。
7. Agent 修改任何任务管理数据前，用户都能看到具体变更并确认或拒绝；工作区文件操作遵循对应工作区写权限和租约。
8. 子任务完成后，父任务满足条件但不被自动完成，父任务 Owner 会收到“可完成”提示。
9. 完成任务后产生可追溯摘要：Agent 完成使用用户自己的模型生成并确认的摘要；人工完成缺少摘要时先提示，继续后使用确定性基本摘要。另一名成员能从平铺树状界面或 Wiki 找到并理解结果。
10. 租约失效后产生的本地改动不会自动上传；重新连接时客户端进入冲突处理，而不是直接覆盖任一侧。
11. 租约或版本冲突不会静默覆盖；仍有写资格的用户明确选择本地或服务端版本后，系统只保留其选择形成的新唯一事实。
12. 已完成任务除评论、显式重新打开和顶层归档外全部冻结；不能给已完成任务新增或移除依赖。
13. 项目按 `deny` 或 `cascade` 策略处理“重新打开前置任务但已有完成 successor”的情况。
14. 关注关系能让 Agent 在关注方任务上下文中无需额外确认地渐进读取被关注任务的授权数据，但不会扩大写权限。
15. 非顶层任务删除前必须输入完整 Task Key，删除后不能恢复且 Task Key 不复用；用户整理过多任务时优先完成并归档顶层任务。
16. Project Owner 可以创建项目级一致备份，并把备份恢复到原项目。
