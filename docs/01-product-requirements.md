# 产品需求说明

文档状态：设计基线 0.3

目标读者：产品设计、客户端开发、服务端开发、Agent/工具开发、测试

相关文档：[领域模型](02-domain-model.md) · [权限模型](03-permission-model.md) · [系统架构](04-system-architecture.md)

## 1. 产品定义

NGAPD 是面向独立游戏开发团队的 AI 原生敏捷项目管理系统。系统使用递归任务表达从项目目标到具体执行工作的完整分解过程，并提供用户级、项目级和任务级三种独立工作区，使人和 Agent 在稳定、分层、可追溯的上下文中协作。

产品不是传统 Jira 的完整替代品，也不是通用网盘或源代码管理系统。它重点解决以下问题：

- 小团队成员往往承担多个专业角色，固定职位无法准确表达能力范围。
- 传统任务系统只保存状态和最终结果，AI 协作产生的大量中间分析无法沉淀。
- Agent 缺少稳定的任务边界，容易混入无关上下文或修改越权对象。
- 项目知识分散在聊天、临时文档和个人目录中，无法随项目演进形成知识库。

## 2. 产品目标

### 2.1 核心目标

- 提供结构简单、可无限递归分解的任务模型。
- 以同级 DAG 表达局部任务执行顺序。
- 以逻辑角色描述成员能力，而非把能力与权限绑定。
- 让用户、项目和任务分别拥有独立的本地/服务端工作区，并形成个人偏好、项目规范和任务过程三个上下文层级。
- 允许 Agent 在权限和人工确认约束下操作项目与任务。
- 将任务过程文档、决策和完成摘要组织为可追溯 Wiki。
- 在少于 20 人的团队中保持低管理成本。

### 2.2 非目标

第一阶段不以以下能力为目标：

- 企业级组织架构、部门、复杂 RBAC/ABAC 策略编辑器。
- 可任意定制的工作流和字段系统。
- 源代码托管、分支管理或 Git 平台替代。
- 通用云盘、大型二进制资产版本控制。
- 工时结算、财务、采购或外包合同管理。
- 多个连接同时写入同一个工作区。
- 无人监督、可绕过确认的自主 Agent。

## 3. 关键术语

| 术语 | 定义 |
|---|---|
| Project Owner | 项目的所有者，唯一；负责项目级配置和管理员任命 |
| Project Admin | 获得项目管理资格的成员；只有主动开启管理员模式后才使用扩大权限 |
| Task Owner | 一个任务唯一的责任人；也是该任务级工作区唯一有资格申请写入租约的成员 |
| 逻辑角色 | 描述成员能力、等级和边界的 Agent 可读文本，不授予系统权限 |
| 任务正文 | 合并任务目标、工作描述和验收信息的单一富文本或 Markdown 字段 |
| 任务管理数据 | 标题、任务正文、截止日期、标签、展示类型、父子关系、依赖、Owner、状态和归档状态等服务端数据 |
| 用户级工作区 | 每个用户一个；对应用户可写，其他已认证用户只读，用于个人流程、自用规则和 Skill |
| 项目级工作区 | 每个项目一个；Project Owner 和 Project Admin 可写，其他项目成员只读，用于项目通用规则、资料和 Skill |
| 任务级工作区 | 每个任务一个；Task Owner 可写，其他项目成员只读，用于任务过程文档和交付物 |
| 逻辑工作区 | 用户、项目或任务创建时在服务端产生的工作区记录与初始内容 |
| 本地工作区 | 具有访问权限的用户首次连接逻辑工作区后，在本地物化的文件目录 |
| 写入租约 | 保证一个工作区同一时间只有一个有效写入会话的机制 |
| 有效状态 | 基础状态结合依赖阻塞、人工阻塞后呈现给用户的状态 |

## 4. 用户类型

### 4.1 项目所有者

- 创建和归档项目。
- 审批成员加入申请。
- 任命或取消 Project Admin。
- 开启管理员模式处理跨 Owner 的任务结构。
- 管理项目级规范和默认逻辑角色。
- 读取并写入项目级工作区。

### 4.2 项目管理员

- 默认仍以普通成员权限工作。
- 显式开启管理员模式后，可以处理任意任务的依赖、Owner、移动和归档。
- 即使开启管理员模式，也不能写入其他 Task Owner 的任务级工作区。
- 无论是否开启管理员模式，都可以读取和写入项目级工作区；项目工作区权限属于独立的工作区权限层。

### 4.3 项目成员

- 维护自己拥有的任务及其任务级工作区。
- 创建、分解、推进和转交任务。
- 读取项目内任务和工作区知识。
- 读取项目级工作区和其他任务工作区；完整操作自己的用户级工作区与自己拥有的任务级工作区。
- 评论任务并接收必要通知。

### 4.4 只读成员

- 查看任务结构、任务详情、评论、完成摘要和允许读取的工作区内容。
- 不得修改当前项目的任务、评论、项目级工作区或任务级工作区；仍可完整操作自己的用户级工作区。

## 5. 核心用户流程

### 5.1 创建项目

1. 用户创建项目并指定全局唯一的 Project Key。
2. 系统复制一份系统预设逻辑角色到项目中。
3. 系统创建项目级工作区、项目级上下文和根任务容器。
4. 创建者成为 Project Owner。
5. Project Owner 可以邀请成员或审批加入申请。

### 5.2 创建和分解任务

1. 成员在顶层或自己有权管理的任务下创建任务。
2. 系统分配不可复用的任务编号，例如 `ABC-123`。
3. 服务端创建任务级逻辑工作区和初始任务文档。
4. Task Owner 可以继续创建子任务、填写统一任务正文并设置截止日期。
5. 同级任务之间可以按照权限建立无环依赖。

### 5.3 Agent 连接任务

1. 用户要求 Agent 连接 `ABC-123`。
2. 系统校验用户身份、项目成员资格和任务读取权限。
3. 如果需要写入任务级工作区，继续校验用户是否为 Task Owner，并获取该工作区的独占写入租约。
4. 本地客户端物化或更新任务目录。
5. 系统把当前用户的用户级工作区规则/Skill、项目级工作区规则/Skill、当前用户在项目中的自我介绍和全部逻辑角色描述、当前任务完整信息、祖先摘要、前置任务摘要和任务级工作区文件提供给 Agent。
6. Agent 在该上下文范围内执行工作，并记录工具操作。

### 5.4 完成任务

1. Task Owner 或其代表的 Agent 发起完成请求。
2. Agent 发起完成时必须获得人工确认。
3. 系统校验全部直接子任务完成、依赖满足、无活动阻塞；条件首次满足时向 Task Owner 发送“可完成”提示。
4. 客户端同步任务级工作区，服务端生成不可变完成快照。
5. 任务进入已完成状态，任务级工作区默认冻结为只读。
6. 后台生成摘要草稿并发布到 Wiki 投影视图。
7. 任务重新打开时创建新的工作周期，保留历史完成快照和摘要。

## 6. 功能需求

### 6.1 账号与项目

- `PRJ-001` 用户可以注册、登录和维护个人资料；注册成功时自动创建唯一用户级逻辑工作区。
- `PRJ-002` 用户可以配置默认逻辑角色和默认自我介绍。
- `PRJ-003` 用户可以创建、打开、归档、恢复和删除项目；创建项目时自动创建唯一项目级逻辑工作区。
- `PRJ-004` Project Key 为 2—6 个大写英文字母，全局唯一且创建后不可修改。
- `PRJ-005` 项目永久删除必须由 Project Owner 在回收站中再次确认。
- `PRJ-006` 项目支持成员申请加入、审批、拒绝和移除。
- `PRJ-007` 项目支持中文项目说明和 Agent 可读项目规范；数据模型预留多语言能力。

### 6.2 逻辑角色

- `ROL-001` 系统内置不可编辑的原始角色模板。
- `ROL-002` 创建项目时复制模板快照；系统模板后续变化不自动覆盖项目角色。
- `ROL-003` 项目内可以创建、编辑、归档和复制逻辑角色。
- `ROL-004` 角色包含名称、等级、能力、责任边界、禁区、任务类型和提示文本。
- `ROL-005` 一个项目成员可以绑定多个逻辑角色。
- `ROL-006` 逻辑角色不得直接授予 Web、工作区或 Agent 权限。

### 6.3 成员

- `MEM-001` Project Owner 可以任命 Project Admin。
- `MEM-002` Project Admin 只有开启管理员模式后才能使用管理权限。
- `MEM-003` 项目成员拥有项目内独立的角色绑定和自我介绍。
- `MEM-004` 新成员加入时从个人默认信息填充项目资料，之后互不自动同步。
- `MEM-005` 每个任务最多一个 Task Owner；允许任务暂时未分配，但开始执行前必须指派。

### 6.4 任务树与依赖

- `TSK-001` 任务可以递归包含子任务，存储层不设置业务深度硬限制。
- `TSK-002` 任务编号在项目内单调分配，归档或删除后不复用。
- `TSK-003` 一个任务只能有一个父任务，且不能移动到自身子树。
- `TSK-004` 依赖只能连接同一父任务的直接子任务；顶层任务统一视为虚拟项目根节点的直接子任务，因此顶层任务之间也可以建立依赖。
- `TSK-005` 同一父任务下的依赖图必须无环。
- `TSK-005A` 不开启管理员模式时，修改依赖要求当前用户拥有两个端点中的至少一个任务，或拥有两者共同的直接父任务；否则必须进入管理员模式。
- `TSK-005B` 顶层任务的共同直接父级是虚拟项目根节点；Project Owner 视为该根作用域的控制者，可以在不开启管理员模式时修改顶层任务之间的依赖。
- `TSK-006` 任务完成的必要条件是全部直接子任务已完成。
- `TSK-007` 子任务完成后父任务不自动完成，必须显式完成。
- `TSK-007A` 父任务满足全部完成条件时，系统将其标记为“可完成”并通知 Task Owner，但不自动改变任务状态。
- `TSK-008` 任务支持未开始、进行中、已阻塞、已完成四种有效状态。
- `TSK-009` 任务只使用截止日期，不在 MVP 中维护预计工时。
- `TSK-010` 归档、移动或删除任务前必须计算对他人任务和依赖的影响。
- `TSK-011` 完成任务下新增子任务前必须重新打开该任务。
- `TSK-012` 任务可以标记为普通、冲刺或里程碑；该标记只改变渲染外观，不参与状态、依赖、完成和权限计算。

### 6.5 初版任务管理界面

- `VIS-001` 初版采用平铺树状导航，不实现语义缩放或嵌套画布。
- `VIS-002` 页面显示当前层级的任务列表，并显示同级任务之间的依赖关系。
- `VIS-003` 页面显示当前选中任务的标题、正文、Owner、状态、截止日期、展示类型和子任务统计。
- `VIS-004` 页面提供任务层级面包屑和“返回上一级”按钮。
- `VIS-005` 页面单独显示当前任务的直接子任务列表，用户可选择子任务进入下一层。
- `VIS-006` 支持按 Owner、逻辑角色、状态、截止日期和标签过滤。
- `VIS-007` 普通、冲刺、里程碑任务使用不同的图标、边框或色彩呈现，但交互和业务规则相同。
- `VIS-008` 支持按 Task Key 或标题搜索并定位到对应层级与任务。
- `VIS-009` 同层 200 个任务时列表、筛选、依赖展示和层级切换仍应保持可用。
- `VIS-010` 语义缩放、节点内嵌子图和无限画布作为后续版本能力，不属于初版验收范围。

### 6.6 工作区

- `WSP-001` 每个用户拥有且仅拥有一个用户级工作区，在用户注册时创建。
- `WSP-002` 每个项目拥有且仅拥有一个项目级工作区，在项目创建时创建。
- `WSP-003` 每个任务拥有且仅拥有一个任务级工作区，在任务创建时创建。
- `WSP-004` 用户级工作区只有对应用户可写，其他已认证用户只读。
- `WSP-005` 项目级工作区只有 Project Owner 和 Project Admin 可写，其他项目成员只读；Project Admin 写入项目级工作区不要求开启管理员模式。
- `WSP-006` 任务级工作区只有当前 Task Owner 可写，其他项目成员只读。
- `WSP-007` 三种工作区均允许多个只读连接，但同一工作区同一时间只能存在一个有效写入租约和一个可写连接。
- `WSP-008` 用户可在用户级工作区保存个人工作流程、自用规则和自用 Skill；可在项目级工作区保存项目通用规则、通用资料和通用 Skill。
- `WSP-009` 三种工作区都以文本为主，支持少量图片和可配置大小限制；服务端保存版本、校验和与快照。
- `WSP-010` 任务层级移动不得改变任务级工作区稳定路径。
- `WSP-011` Task Owner 转移前必须同步并释放任务级工作区写入租约。
- `WSP-012` 已完成任务的任务级工作区默认冻结；项目级和用户级工作区不随单个任务完成而冻结。

### 6.7 Agent

- `AGT-001` Agent 工具与 Web UI 使用相同的服务端领域规则。
- `AGT-002` Agent 操作必须以当前用户身份执行，不拥有独立的越权身份。
- `AGT-003` Agent 必须同时通过 Web 权限、工作区权限和确认策略。
- `AGT-004` Agent 可以读取当前用户有权查看的全部任务管理信息，读取不需要人工确认。
- `AGT-005` Agent 对当前用户有写权限且已取得租约的用户级、项目级或任务级工作区具有完整访问权限，可以创建、修改、移动、重命名和删除其中的文件。
- `AGT-006` Agent 对任务管理数据的任何修改都必须先形成操作提案并获得人工确认，包括修改标题或任务正文、创建任务、评论、状态、依赖、Owner、层级、展示类型、归档和完成。
- `AGT-007` Agent 必须显式进入管理员模式才能使用管理员权限；进入动作只能由用户确认触发，Agent 不能自动进入。
- `AGT-008` Agent 可以主动退出管理员模式，也应在任务完成、会话结束或不再需要管理权限时自动退出。
- `AGT-009` 所有 Agent 写操作必须记录输入、操作者、确认结果和关联任务；工作区文件操作同样保留本地/同步审计。

### 6.8 评论、通知与知识库

- `KNW-001` 项目成员可以对任务发表评论；评论不改变任务和工作区权限。
- `KNW-002` 支持 Owner 变更、被指派、阻塞、依赖变化、评论和截止日期提醒。
- `KNW-002A` 父任务达到“可完成”条件时通知其 Task Owner，并对同一完成条件版本去重。
- `KNW-003` 用户可以配置非关键通知偏好，权限和破坏性操作通知不可关闭。
- `KNW-004` 完成任务后生成带来源链接的摘要草稿。
- `KNW-005` Wiki 按任务树、角色、标签和全文搜索组织知识。
- `KNW-006` 重新完成任务时保留旧摘要和旧工作区快照。
- `KNW-007` Wiki 页面不能脱离任务来源静默覆盖历史内容。

## 7. 非功能需求

### 7.1 平台与部署

- 本地客户端：macOS、Windows。
- 服务端：Linux，自托管优先。
- Agent 协议和摘要模型应保持供应商无关。

### 7.2 目标规模

- 典型项目成员少于 20 人。
- 典型同层任务少于 200 个，不作为业务硬限制。
- 工作区以文本为主，支持少量图片。
- 系统应使用分页、懒加载和当前层依赖渲染，不能依赖上述规模作为正确性前提。

### 7.3 一致性与审计

- 任务更新使用版本号进行乐观并发控制。
- 结构修改和 Agent 操作必须可审计。
- 写入租约和 Owner 转移必须原子化。
- 关键写操作提供幂等键，避免重试产生重复任务或依赖。

### 7.4 安全

- 本地客户端只能访问用户明确配置的 NGAPD 工作区根目录及其中已登记的用户级、项目级和任务级目录。
- Agent 工具不能通过路径穿越访问工作区外文件。
- 服务端对象存储路径不得直接使用用户输入。
- 工作区文件视为不可信内容，加载给 Agent 时必须与系统指令分隔。

## 8. MVP 范围

MVP 必须形成完整闭环，而不是只实现通用任务 CRUD：

1. 账号、项目、成员申请和简单权限。
2. 系统角色模板、项目角色和成员多角色绑定。
3. 递归任务、单 Owner、截止日期、状态和同级 DAG。
4. 平铺树状任务界面：本级任务列表、当前任务详情、层级面包屑、直接子任务列表和返回上一级按钮。
5. 用户级、项目级、任务级逻辑工作区，本地物化、每工作区单写租约和文本/图片同步。
6. Agent 连接任务、读取全部可见任务信息和分层工作区上下文，完整操作当前用户有写权限且持有租约的工作区，并通过人工确认修改任何任务管理数据。
7. 完成快照、摘要草稿、基础 Wiki 浏览和搜索。
8. 评论、被指派通知和关键审计记录。

MVP 暂缓：语义缩放、嵌套任务图和无限画布、复杂通知规则、Git 平台深度集成、多人工作区编辑、可定制工作流、语义向量检索和移动客户端。

## 9. 产品级验收标准

当一个由策划、程序和美术组成的小团队能够完成以下流程时，MVP 才算成立：

1. 建立项目、角色和成员。
2. 把一个游戏功能拆成多层任务，并建立无环同级依赖。
3. 用户可以在用户级工作区保存个人流程，在项目级工作区保存团队规则，并在任务级工作区保存过程文档。
4. Project Admin 可以写项目级工作区但不能写他人的任务级工作区；其他项目成员只读项目级工作区。
5. 三种工作区在出现第二个写入连接时都拒绝授予新租约或要求显式接管。
6. Agent 在当前有写权限并取得租约的工作区中完整操作文件，且不能写入没有写权限的工作区。
7. Agent 修改任何任务管理数据前，用户都能看到具体变更并确认或拒绝；工作区文件操作遵循对应工作区写权限和租约。
8. 子任务完成后，父任务满足条件但不被自动完成，父任务 Owner 会收到“可完成”提示。
9. 完成任务后产生可追溯摘要，另一名成员能从平铺树状界面或 Wiki 找到并理解结果。
