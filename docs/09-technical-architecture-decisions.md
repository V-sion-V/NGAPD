# 技术架构决策

文档状态：MVP 技术基线 0.1

相关文档：[系统概要设计](04-system-architecture.md) · [实施路线](07-roadmap-and-validation.md) · [决策记录](08-decisions-and-open-issues.md) · [MVP 非功能基线](10-mvp-non-functional-baseline.md)

## 1. 决策上下文

MVP 初期面向少于 10 名用户的小规模独立游戏团队，通过内网或 VPN 访问一台 Linux 服务器。优先级依次为：

1. 尽快交付完整业务闭环。
2. 保证领域事务、工作区单写入和备份恢复正确。
3. 保持单机部署、升级和排错简单。
4. 在不引入额外常驻基础设施的前提下保留替换边界。

当前阶段不为公网大规模访问、高并发、高可用集群、跨地域容灾或多租户商业 SaaS 提前建设基础设施。所有业务数据自托管；NGAPD 服务端运行时不调用外部 API、AI 或 LLM。

## 2. ADR-001：代码库与主要语言

**决策**

- 使用一个 Git 单仓库和 `pnpm workspace`。
- Web、服务端、后台作业、共享契约和 Agent 工具服务主要使用 TypeScript。
- Electron 主进程和渲染界面使用 TypeScript；只有确有必要的操作系统能力才增加小型原生模块。
- API Schema、错误码、领域枚举和 Agent 工具输入输出放入共享包，但领域实现不共享到客户端。

**理由**

- 减少小团队跨语言切换和重复模型维护。
- 允许 Web、桌面端和工具服务复用经过生成或校验的契约类型。
- 保持模块化单体的部署和本地调试简单。

**约束**

- 共享类型不能替代运行时校验。
- 客户端不能导入服务端领域服务或数据库代码。
- 依赖版本在 M0 工程骨架中通过 lockfile 固定。

## 3. ADR-002：服务端框架与模块化单体

**决策**

- Node.js TypeScript 服务端使用 Fastify。
- 以业务模块组织代码：Identity、Projects/Membership、Roles、Tasks、Dependency Graph、Authorization/Audit、Workspaces、Agent Operations、Knowledge/Notifications。
- 模块通过应用服务、只读查询接口或领域事件交互；禁止跨模块直接修改数据表。
- HTTP 应用与后台 Worker 使用同一代码库和领域包，但作为两个独立进程运行。

**理由**

- Fastify 提供较小运行时开销、成熟的 JSON Schema/OpenAPI 集成和直接的请求生命周期。
- 不引入额外分布式服务，保留数据库事务协调能力。

**禁止**

- 不拆微服务。
- 不允许通用 Repository 绕过模块归属。
- 不通过消息队列承担必须同步提交的领域不变量。

## 4. ADR-003：关系数据库和数据访问

**决策**

- 使用 PostgreSQL 作为唯一权威业务数据库。
- TypeScript 数据访问使用 Kysely；迁移使用受版本控制的 SQL/Kysely migration。
- 递归任务读取使用递归 CTE。
- DAG、移动、完成、Owner/租约协调使用显式事务、行锁和稳定锁顺序。
- 关键不变量同时落实为数据库唯一约束、外键、检查约束和应用服务校验。

**理由**

- 当前领域高度依赖事务、唯一约束、递归查询、行锁和一致备份。
- 单个数据库足以覆盖初期规模，并减少 Redis、分布式锁和跨服务事务。

**实现要求**

- 每个迁移都提供前向执行检查；破坏性迁移必须先备份。
- M0 建立事务并发测试，验证 `graph_version`、Owner 变化、租约和完成冻结。
- 所有租户查询必须携带服务端解析的 `project_id`。

## 5. ADR-004：文件与对象存储

**决策**

- MVP 在同一 Linux 主机上使用持久 Docker Volume 保存内容寻址对象。
- 对象使用 SHA-256 内容哈希和服务端内部 ID 定位；用户路径只存在于 PostgreSQL 文件清单中。
- 在代码中定义 `ObjectStore` 接口，首个实现为本地文件存储；未来需要时可以增加 S3 兼容实现。
- 初期不部署 MinIO、Ceph 或云对象存储。

**理由**

- 当前工作区以文本和少量图片为主，单机文件存储足够。
- 内容寻址和存储接口保留未来迁移能力，同时避免增加一个需要备份和监控的服务。

**约束**

- 对象目录不能直接暴露给用户或 Agent。
- 数据库清单与对象内容必须共同进入项目备份和恢复校验。
- 垃圾回收只能删除已经不存在任何版本引用的对象。

## 6. ADR-005：搜索

**决策**

- MVP 使用 PostgreSQL Full Text Search 和 `pg_trgm` 完成任务、摘要和允许索引文本的搜索。
- 搜索索引是可重建投影，不作为业务权威状态。
- 不部署 Elasticsearch、OpenSearch 或向量数据库。

**升级触发条件**

- PostgreSQL 搜索在 MVP 数据规模下不能满足轻量非功能基线。
- 出现跨项目大规模检索、复杂相关性或独立扩容要求。

## 7. ADR-006：HTTP API、实时事件与契约

**决策**

- 对外业务 API 使用版本化 REST/JSON。
- 使用 TypeBox JSON Schema 进行运行时校验并生成 OpenAPI 3.1 文档。
- 乐观并发通过资源 `version`、`graph_version` 和必要的条件请求实现。
- 创建、提案执行、同步提交和后台作业入口支持幂等键。
- 实时通知使用 SSE；业务提交仍通过 HTTP。
- 桌面客户端定时续租使用普通 HTTP，不依赖实时通道维持租约。

**理由**

- REST/OpenAPI 易于调试、生成客户端并供 Agent 工具服务调用。
- 当前实时需求主要是通知客户端重新获取资源，SSE 比双向 WebSocket 更简单。

**错误契约**

错误响应至少包含稳定机器码、用户可读信息、关联请求 ID、当前版本和可执行修复建议。

## 8. ADR-007：后台作业

**决策**

- 使用 PostgreSQL 持久作业表；TypeScript Worker 首选 Graphile Worker。
- 用于通知投影、全文索引、备份校验和清理等可重试异步工作。
- 不引入 Redis、RabbitMQ 或 Kafka。
- NGAPD 不创建调用外部模型的摘要作业。

**可靠性规则**

- 作业处理至少一次，处理器必须幂等。
- 业务事务通过 Outbox 记录待投影事件，提交后由 Worker 消费。
- 作业失败保留原因和重试次数；达到上限后进入可人工重试状态。

## 9. ADR-008：Web 前端

**决策**

- 使用 React、Vite 和 TypeScript。
- 服务端 OpenAPI 生成基础 API Client，前端查询缓存使用 TanStack Query。
- 当前层任务列表使用虚拟化；依赖只布局当前父级作用域。
- UI 组件首先满足中文界面、键盘可达性和颜色之外的状态区分。

**暂不采用**

- 服务端渲染框架。
- 无限画布和递归嵌套图。
- 独立前端微应用。

## 10. ADR-009：桌面客户端与同步

**决策**

- 使用 Electron + React 构建 Windows/macOS 客户端。
- Electron 主进程负责认证凭证访问、目录登记、文件监听、哈希、原子替换、租约和同步；渲染进程不直接获得任意文件系统能力。
- 文件监听首选 Chokidar，提交以重新计算的 SHA-256 清单为准，不依赖监听事件本身保证正确性。
- 本地数据库使用 SQLite，只保存设备配置、已登记工作区、本地 manifest、同步状态和非权威队列；服务端业务数据不在本地成为权威状态。
- 凭证使用 Electron `safeStorage` 和操作系统凭证能力加密保存。

**跨平台规则**

- Windows 11 x64 是 Windows 首要验收平台。
- macOS 在 M4 冻结时支持当时最近两个主要版本，Apple Silicon 为首要验收架构；Intel 构建根据团队实际设备保留。
- 服务端清单路径使用 `/` 分隔、UTF-8 和 Unicode NFC；客户端在写入前检查大小写碰撞、保留名称和非法字符。
- 符号链接默认不跟随、不上传。

## 11. ADR-010：本地 Agent 工具与 IPC

**决策**

- 工具协议采用 MCP 风格的结构化工具契约。
- Agent 工具服务作为桌面客户端安装包的一部分运行，不持有独立越权身份。
- Agent 宿主连接优先使用标准输入输出；需要常驻连接时，macOS 使用 Unix Domain Socket，Windows 使用 Named Pipe。
- 本地连接必须绑定当前 OS 用户、桌面客户端会话和短期随机能力；不得监听局域网端口。
- 工具服务只能通过桌面客户端登记的 Workspace ID 访问规范化路径。

**复核点**

本 ADR 固定边界和默认方向；OS ACL、宿主配对、能力续期和具体凭证格式仍按 D-064 在 M5 前进行安全复核。

## 12. ADR-011：身份认证

**决策**

- MVP 提供本地账号和密码认证，不接入外部身份提供商。
- 密码使用 Argon2id 哈希。
- Web 使用 Secure、HttpOnly、SameSite Cookie 会话。
- 桌面客户端使用短期访问令牌和可撤销设备凭证；长期材料只保存于 OS 凭证保护区。
- 管理员模式能力与普通登录会话分离，30 分钟无操作后失效。

**部署要求**

- 即使位于 VPN 内，HTTP 仍通过反向代理提供 TLS。
- MVP 默认不要求每次进入管理员模式重新认证；高风险部署可以启用。

## 13. ADR-012：摘要和外部服务边界

**决策**

- NGAPD 服务端和 Worker 不调用外部 API、AI 或 LLM。
- 用户通过 Agent 完成任务时，用户自己的 Agent/模型生成摘要，并把摘要正文、来源清单和来源类型随完成提案提交。
- 人工完成没有摘要时，界面先提示；用户继续后，服务端使用固定模板和基本任务字段生成一句话摘要。
- 用户 Agent 的模型选择、费用、密钥和网络访问属于 Agent 宿主，不属于 NGAPD 配置或数据。

**数据要求**

KnowledgeEntry 记录 `agent_provided / user_provided / system_fallback` 来源，不把 Agent 摘要描述成服务端 AI 生成。

## 14. ADR-013：单机部署

**决策**

Docker Compose 至少包含：

- `gateway`：Caddy，提供 TLS、静态 Web 和反向代理。
- `app`：Fastify HTTP/SSE 应用。
- `worker`：后台作业和投影。
- `postgres`：业务数据库和作业表。
- 持久卷：PostgreSQL 数据、内容寻址对象、备份、应用日志。

MVP 不部署 Kubernetes、Redis、独立消息队列、独立搜索服务或独立对象存储服务。

**运维方式**

- 通过版本化镜像和 Compose 文件发布。
- 数据库迁移在应用切换前显式执行。
- 升级前创建数据库和对象卷备份；失败时回退应用镜像并按迁移说明处理数据库。
- 健康检查、结构化日志和备份校验是单机部署的最低运维能力。

## 15. M0 仍需验证的实现细节

以下事项不改变核心选型，但应在 M0 工程骨架中用小型 Spike 固定：

- Fastify、TypeBox、Kysely 和 OpenAPI 生成链路。
- PostgreSQL 递归 CTE、稳定行锁顺序和并发测试夹具。
- Graphile Worker 与 Outbox 的事务交接。
- 本地内容存储、清单哈希和项目备份的一致检查点。
- Electron 文件监听、原子替换、SQLite 状态和安全路径 API。
- SSE 断线重连、事件游标和客户端重新获取资源。

验证失败时新增 ADR 记录替代选择，不直接静默改变本文件的既有决定。
