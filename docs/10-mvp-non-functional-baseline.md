# MVP 非功能基线

文档状态：交付优先基线 0.1

相关文档：[产品需求](01-product-requirements.md) · [技术架构决策](09-technical-architecture-decisions.md) · [实施路线](07-roadmap-and-validation.md)

## 1. 原则

本基线只防止 MVP 出现明显不可用、数据丢失或无法恢复的问题，不以高并发、高可用或大规模容量建设阻塞交付。

优先级为：

1. 领域不变量和权限正确。
2. 工作区不静默覆盖，项目备份能够恢复。
3. 少于 10 名用户在内网或 VPN 中使用流畅。
4. 其他优化在真实数据证明需要后再实施。

除明确标注为业务规则的项目外，本文件数值都是初期验收目标或部署默认值，不是产品硬限制。

## 2. 初期负载假设

| 项目 | 初期目标 |
|---|---|
| 注册用户 | 少于 10 |
| 同时活跃用户 | 通常少于 5，按 10 验收 |
| 单项目活动任务 | 通常少于 2,000，按 5,000 验收 |
| 单父级直接子任务 | 典型少于 200，按 200 验收；不是硬限制 |
| 任务树深度 | 典型少于 8 层，按 20 层验证递归正确性；不是硬限制 |
| 单工作区受管文件 | 典型少于 500，按 2,000 验收 |
| 单文件默认软限制 | 50 MiB，可由部署配置调整 |
| 单工作区默认软限制 | 2 GiB，可由部署配置调整 |

超过软限制时应给出明确提示并阻止新的超限同步，不得导致现有服务数据损坏。

## 3. 参考服务器

最低参考配置：

- Linux x86-64。
- 4 个 CPU 核心。
- 8 GiB 内存。
- 100 GiB SSD 可用空间。
- Docker Engine 和 Docker Compose。
- 通过内网或 VPN 访问。

建议配置为 4 个 CPU 核心、16 GiB 内存和至少 200 GiB SSD。磁盘空间主要由工作区对象、备份和日志决定。

不要求双机、集群、自动故障转移或跨地域复制。

## 4. 交互与 API 目标

在参考服务器、初期负载和正常内网/VPN 条件下：

| 操作 | MVP 目标 |
|---|---|
| 当前层任务列表、任务详情 | 服务端 P95 小于 500 ms |
| 创建或更新普通任务 | 服务端 P95 小于 800 ms |
| 200 节点局部 DAG 读取 | 服务端 P95 小于 800 ms |
| 任务/摘要全文搜索 | P95 小于 1 s |
| Agent 上下文清单 | P95 小于 1 s，不含按需文件下载 |
| Web 首次可交互 | 暖缓存内网环境小于 3 s |
| 页面层级切换 | 数据返回后 200 ms 内完成主要界面更新 |

这些目标不要求单独建设缓存集群。未达标时先通过索引、查询、分页和客户端渲染修正。

## 5. 工作区同步目标

- 500 个文本为主的文件、总计 100 MiB 的已物化工作区，增量扫描和差异计算在参考桌面设备上应在 5 秒内给出状态。
- 已知新增或修改的小文件应在网络可用且租约有效时，于 10 秒内进入服务端新版本。
- 初次物化和大图片上传只显示明确进度，不设置会阻塞 MVP 的统一硬时限。
- 任一租约或版本冲突都不得静默覆盖。
- 失去写资格的客户端不得把旧本地副本提交为服务端事实。

租约默认参数在 M4 同步原型后冻结；初始测试值为：

- TTL：60 秒。
- 续租间隔：20 秒。
- 断网宽限：不超过剩余 TTL。
- 强制接管：原持有者通知后可由合格用户立即确认执行，不增加固定等待时间。

## 6. 可用性和恢复

- MVP 不承诺正式 SLA；计划维护和单机故障期间可以停机。
- 应用或 Worker 重启不能破坏已提交数据库事务和服务端工作区版本。
- 通知、索引等投影允许最终一致，并可从数据库/Outbox 重建。
- 建议每天备份 PostgreSQL 和对象卷，默认保留最近 7 个日备份；具体计划由部署者配置。
- 初期运维目标：RPO 不超过 24 小时，RTO 不超过 4 小时。
- 每个发布候选至少完成一次项目级备份完整性校验；涉及备份格式或恢复逻辑的版本必须执行原项目恢复演练。

项目级应用备份仍必须满足数据库清单和对象内容一致性，不能用普通文件复制替代。

## 7. 安全最低基线

- 仅通过内网或 VPN 暴露服务，不直接开放数据库、对象目录或本地 Agent IPC。
- Web/API 即使在 VPN 内仍使用 TLS。
- 密码使用 Argon2id；令牌和密码不写入普通日志。
- Web 使用 Secure、HttpOnly、SameSite Cookie。
- 桌面长期凭证保存在操作系统凭证保护区。
- Docker 配置中的密钥通过部署环境或 Docker Secret 提供，不提交到 Git。
- 所有任务写入、管理员模式、租约接管、Agent 提案、备份和恢复保留审计。
- NGAPD 服务端运行时不调用外部 API、AI 或 LLM，也不向外部服务发送业务内容。
- 工作区内容、评论和角色提示按不可信输入处理，工具执行始终重新授权。

MVP 不要求建设 SIEM、WAF、硬件密钥管理或独立安全监控集群。

## 8. 客户端兼容矩阵

必须验收：

- Windows 11 x64。
- M4 冻结时最近两个主要版本的 macOS；Apple Silicon 为首要架构。
- NTFS 和 macOS 默认文件系统上的大小写、Unicode、保留名称、长路径和符号链接场景。

Intel Mac 安装包根据团队实际设备决定；若提供，必须运行同一套工作区契约测试。Windows 10 和 Linux 桌面客户端不属于 MVP 保证范围。

## 9. 日志与可观测性

最低实现：

- JSON 结构化应用日志，包含时间、级别、请求 ID、模块和稳定错误码。
- `/health/live` 和 `/health/ready`。
- 后台作业状态、失败原因和人工重试入口。
- PostgreSQL、对象卷和备份目录磁盘空间检查。
- 审计日志与普通诊断日志分离。

初期不要求 Prometheus、Grafana、集中日志平台或分布式追踪。出现实际排障需求后再增加。

## 10. 发布门槛

非功能门槛只包含：

1. 参考负载下核心页面和 API 没有明显卡顿或超时。
2. 两个客户端不能形成两个合法写入流。
3. 租约、版本和路径异常不会静默覆盖服务端事实。
4. 服务重启后已提交数据和工作区版本保持一致。
5. 项目备份通过完整性检查并能恢复到原项目。
6. Windows 11 和目标 macOS 设备通过工作区契约测试。
7. 运行时网络检查确认 NGAPD 服务端没有外部 API 或 LLM 依赖。

未达到其他建议指标时可以记录为后续优化，不因此扩大 MVP 基础设施范围。
