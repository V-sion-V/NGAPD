## 2. 权限解释基线

用户给出的原则同时包含“可以管理自己节点及其子节点”和“不能影响别人的任务”。为了避免父任务 Owner 获得隐式管理链，本版采用更严格解释：

- 父任务 Owner 可以创建子任务，并管理尚未指派、由自己的最近祖先所辖的子任务。
- 子任务一旦指派给其他成员，父任务 Owner 只能读取它，不能编辑、改状态、归档或写其任务级工作区。
- 涉及他人任务的子树移动、顶层归档、非顶层删除等结构操作需要管理员模式。
- 依赖增删采用双端责任规则：拥有两端或共同直接父任务可以直接修改；只拥有一个端点时由另一端点 Owner 接受；顶层由 Project Owner 控制。
- 管理员模式仍然不能写他人的任务级工作区；项目级工作区由 Project Owner/Admin 根据独立写入策略和租约访问。

这能保持“每个任务只有一个人负责”在权限上也成立。

## 3. Agent 确认解释基线

当前文档把 Agent 操作划分为：

- 无需确认：读取全部授权任务管理信息；读取默认分层工作区、项目角色和知识；在当前用户有写资格且已获得租约的用户级或任务级工作区中创建、修改、移动、重命名和删除用户内容；正常同步；退出管理员模式。
- 需要明确用户意图：读取其他用户的用户级工作区；Agent 不得自行发现、搜索或加载。
- 需要显式管理员模式：Agent 写入项目级工作区；必须由用户先明确要求本次项目维护并确认进入。
- 需要确认：任何任务管理数据修改，包括标题、统一正文、UTC 截止时间、逻辑角色、标签、展示类型、评论、任务创建/拆分、状态、依赖、Owner、层级、完成和重新打开。
- 需要明确影响预览和确认：进入管理员模式、跨 Owner 修改、任务归档、批量修改、强制租约接管和永久删除。

人工确认不能替代权限；无权操作即使确认也应拒绝。

## 5. 需要特别验证的假设

### H-001：三级工作区边界足够稳定

假设：用户级、项目级、任务级分层，加上按 Task Key 平铺的任务目录，能让 Agent 保持上下文隔离，同时不会给成员带来过高的目录管理成本。

验证：让真实任务连续使用多个工作周期，观察跨任务引用、搜索和人工整理成本。

### H-002：摘要能替代大量原始过程读取

假设：用户级个人流程、项目级通用规则、活动祖先当前信息，以及 ancestor/predecessor 的历史确认摘要足以为 Agent 提供必要背景，不需要加载所有工作区完整原文。

验证：比较只用摘要与加载原文时的任务理解和完成质量。

### H-003：同级依赖限制符合实际工作

假设：跨父任务依赖可以通过在共同父任务下提升/增加协调节点表达，不需要任意跨树边。

验证：记录真实项目中无法自然表达的依赖案例；在放宽限制前优先改进建模指引。

### H-004：三层权限足够简单

假设：管理员模式、三级工作区独占写入和 Agent 确认模型既能防误操作，又不会给小团队造成明显负担。

验证：统计管理员模式使用频率、被拒绝但用户认为合理的操作，以及 Agent 确认接受/拒绝率。

### H-005：平铺树状导航足以支撑初版

假设：本级任务列表、当前任务详情、面包屑、子任务列表和返回上一级按钮，足以清晰表达初版的任务分解与同级依赖。

验证：让用户使用平铺树状界面完成深层任务定位、顶层/同级依赖分析和进度理解；记录必须依赖语义缩放才能解决的问题，作为后续版本输入。

## 6. 变更规则

- 已确认决策发生变化时，新增一条决策记录，不直接删除旧结论。
- 领域不变量变化必须同步更新产品需求、领域模型、权限模型、Agent 工具和测试计划。
- 权限放宽必须给出具体威胁分析和回归测试。
- 工作区格式通过 `schemaVersion` 演进，客户端必须能够识别不支持的新版本并安全降级为只读。
