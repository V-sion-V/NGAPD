## 2. 权限解释基线

用户给出的原则同时包含“可以管理自己节点及其子节点”和“不能影响别人的任务”。为了避免父任务 Owner 获得隐式管理链，本版采用更严格解释：

- 父任务 Owner 可以创建子任务，并管理尚未指派、由自己的最近祖先所辖的子任务。
- 子任务一旦指派给其他成员，父任务 Owner 只能读取它，不能编辑、改状态、归档或写工作区。
- 涉及他人任务的子树移动、归档和依赖影响需要管理员模式。
- 管理员模式仍然不能写他人的工作区。

这能保持“每个任务只有一个人负责”在权限上也成立。

## 3. Agent 确认解释基线

当前文档把 Agent 操作划分为：

- 无需确认：读取；修改当前连接且由用户拥有的任务描述性内容；在已获得租约的当前工作区中进行非破坏性过程文档写入；正常同步。
- 需要确认：任务创建/拆分、状态、依赖、Owner、层级、评论、完成和重新打开。
- 需要影响预览和确认：归档、文件删除、批量覆盖、强制租约接管、管理员模式跨 Owner 操作。

人工确认不能替代权限；无权操作即使确认也应拒绝。

## 5. 需要特别验证的假设

### H-001：任务级工作区足够稳定

假设：按 Task Key 平铺的独立工作区能让 Agent 保持上下文隔离，同时不会给成员带来过高的目录管理成本。

验证：让真实任务连续使用多个工作周期，观察跨任务引用、搜索和人工整理成本。

### H-002：摘要能替代大量原始过程读取

假设：祖先和 predecessor 的确认摘要足以为 Agent 提供必要背景，不需要加载其完整工作区。

验证：比较只用摘要与加载原文时的任务理解和完成质量。

### H-003：同级依赖限制符合实际工作

假设：跨父任务依赖可以通过在共同父任务下提升/增加协调节点表达，不需要任意跨树边。

验证：记录真实项目中无法自然表达的依赖案例；在放宽限制前优先改进建模指引。

### H-004：三层权限足够简单

假设：管理员模式、Task Owner 单写和 Agent 确认三层模型既能防误操作，又不会给小团队造成明显负担。

验证：统计管理员模式使用频率、被拒绝但用户认为合理的操作，以及 Agent 确认接受/拒绝率。

### H-005：语义缩放优于传统树形列表

假设：嵌套图能更好表达独立游戏开发中的任务分解与依赖，而不会使导航复杂化。

验证：让用户分别用树形视图和语义缩放图完成定位、阻塞分析和进度理解任务。

## 6. 变更规则

- 已确认决策发生变化时，新增一条决策记录，不直接删除旧结论。
- 领域不变量变化必须同步更新产品需求、领域模型、权限模型、Agent 工具和测试计划。
- 权限放宽必须给出具体威胁分析和回归测试。
- 工作区格式通过 `schemaVersion` 演进，客户端必须能够识别不支持的新版本并安全降级为只读。
