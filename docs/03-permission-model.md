# 权限与安全模型

文档状态：设计基线 0.1  
相关文档：[领域模型](02-domain-model.md) · [Agent 设计](06-agent-integration.md) · [工作区设计](05-workspace-context-wiki.md)

## 1. 权限模型总览

系统存在三个必须分离的权限层：

1. **Web/任务权限**：决定用户能否修改项目数据、任务内容和任务结构。
2. **工作区权限**：决定用户能否读取或写入任务绑定的文件工作区。
3. **Agent 权限与确认**：Agent 先继承当前用户权限，再根据操作风险要求人工确认。

任何后一层都不能扩大前一层授予的权限。最终允许条件为：

```text
allow = authenticated
    AND active_project_membership
    AND web_task_authorized
    AND workspace_authorized_if_needed
    AND agent_confirmation_satisfied_if_needed
```

管理员模式只影响第一层。Project Owner 或 Project Admin 即使开启管理员模式，也不能写入其他 Task Owner 的工作区。

## 2. 角色与能力分离

### 2.1 系统权限角色

| 权限角色 | 说明 |
|---|---|
| Project Owner | 每项目唯一，拥有项目生命周期、成员审批和管理员任命权 |
| Project Admin | 有资格开启管理员模式，但默认仍按普通成员权限操作 |
| Member | 可以创建任务并管理自己的任务与工作区 |
| Viewer | 只读项目、任务、评论、工作区快照和 Wiki |

### 2.2 逻辑角色

“程序员 L1”“角色原画 L2”等逻辑角色只描述能力。授权服务不得根据逻辑角色名称、等级、提示词或标签放行操作。

这意味着“程序员 L3”不会自动获得管理员权限，“项目管理员”也不会自动被视为某个专业角色。

## 3. 管理员模式

### 3.1 设计目的

Project Admin 的资格和管理员模式的启用状态分开，防止管理员在日常工作时误改其他成员的任务。

### 3.2 启用规则

- 只有 Project Owner 和活动状态的 Project Admin 可以启用。
- 必须由用户在界面中主动开启，Agent 无权代替用户开启。
- 开启后显示持续、明显的视觉提示。
- 使用短期管理会话，建议 30 分钟无操作后失效。
- 高风险部署可以要求重新验证密码或二次认证。
- 开启、关闭和过期均写入安全审计。

### 3.3 能力范围

开启管理员模式后，可以：

- 修改任意任务内容和状态。
- 修改任意合法的同级依赖。
- 修改任意任务 Owner。
- 移动、归档、恢复任务。
- 处理跨 Owner 影响的任务子树。
- 管理项目角色、成员权限和项目配置（仍受 Project Owner 专属操作限制）。

管理员模式不能：

- 写入其他 Task Owner 的工作区。
- 静默覆盖正在同步的工作区。
- 跳过 Agent 人工确认。
- 修改不可变 Task Key 或历史审计。
- 代替 Project Owner 永久删除项目或任命新的 Project Owner。

## 4. 普通成员的任务权限

### 4.1 基本规则

普通模式下，成员只管理自己拥有的任务，以及自己有权管理的未分配子任务。父任务 Owner 不自动获得已分配给他人的后代任务内容权限。

“可以管理自己的节点及其子节点”采用以下保守含义：

- 可以在自己的任务下创建子任务。
- 可以继续管理尚未分配、且最近已分配祖先由自己拥有的子任务。
- 子任务一旦指派给其他成员，其内容、状态和工作区即属于新 Owner 的权限范围。
- 父任务 Owner 可以读取后代状态和摘要，但不能修改其他 Owner 的节点。
- 任何会改变其他 Owner 任务可执行性、Owner、层级或存续状态的操作，都需要管理员模式。

### 4.2 任务权限矩阵

| 操作 | 普通 Member | 管理员模式 | Viewer |
|---|---|---|---|
| 查看项目任务 | 允许 | 允许 | 允许 |
| 创建顶级任务 | 允许，但必须指定 Owner | 允许，可未分配 | 禁止 |
| 在自己的任务下创建子任务 | 允许 | 允许 | 禁止 |
| 在他人任务下创建子任务 | 禁止 | 允许 | 禁止 |
| 修改自己任务的描述性内容 | 允许 | 允许 | 禁止 |
| 修改他人任务的描述性内容 | 禁止 | 允许 | 禁止 |
| 修改自己任务状态 | 允许，仍受状态机约束 | 允许，仍受状态机约束 | 禁止 |
| 修改他人任务状态 | 禁止 | 允许 | 禁止 |
| 把自己的任务转交给他人 | 允许；转交后立即失去写权限 | 允许 | 禁止 |
| 取得他人任务 Owner | 禁止 | 允许 | 禁止 |
| 移动自己的任务 | 仅当源、目标和影响集合均不涉及他人任务 | 允许 | 禁止 |
| 归档自己的任务 | 仅当影响集合均在自己权限内 | 允许 | 禁止 |
| 归档含他人任务的子树 | 禁止 | 允许并展示影响预览 | 禁止 |
| 永久删除任务历史 | 禁止 | 禁止；仅 Project Owner 回收站操作 | 禁止 |

### 4.3 未分配任务

- 未分配任务不能进入进行中或已完成。
- 自己拥有的任务下，未分配子任务由该最近祖先 Owner 暂时管理。
- 一旦未分配任务被指派，其管理权转移给新 Task Owner。
- 为避免产生无管理者的顶级节点，普通成员创建顶级任务时必须指定 Owner。
- 管理员模式可以创建和整理顶级未分配任务。

`created_by` 仅用于审计，不把创建者变成第二 Task Owner。

## 5. 依赖权限

依赖 `predecessor -> successor` 直接改变 successor 是否可以开始，因此普通模式下以 successor 的 Owner 作为主要授权对象。

### 5.1 普通模式允许

- Task Owner 可以为自己的 successor 添加或移除一个合法的同级 predecessor。
- predecessor 可以由自己或其他成员拥有，因为该边只阻塞自己的 successor。
- 两端必须同父，操作后仍必须无环。

### 5.2 普通模式禁止

- 以他人的任务为 successor 添加或删除依赖。
- 通过归档、移动 predecessor 间接解除他人 successor 的依赖。
- 批量重排包含他人 successor 的依赖图。

后一类操作需要管理员模式。系统必须基于“受影响 successor 集合”授权，而不是只检查用户是否拥有被点击的节点。

### 5.3 依赖变更请求

MVP 可以由管理员处理跨 Owner 依赖。后续可增加“依赖变更请求”：成员提出建议，由 successor Owner 接受后生效。该请求不是依赖边本身，也不改变任务状态。

## 6. 影响集合授权

移动、归档、恢复和 Owner 转移不能只检查目标任务。领域服务必须先计算影响集合：

- 目标任务及其全部后代。
- 将被删除或失效的依赖边。
- 因依赖变化而改变有效状态的 successor。
- 已完成祖先的完成合法性。
- 活动工作区写入租约和未同步版本。

普通模式要求影响集合内所有需要写入的已分配任务都由当前用户拥有；否则拒绝并列出需要管理员处理的原因。

## 7. 工作区权限

### 7.1 读取

- 活动项目成员和 Viewer 可以读取项目内工作区已同步版本。
- 已被移除的成员立即失去服务端读取权限。
- 本地已经下载的副本无法通过服务端远程擦除，客户端应提示其权限已失效并停止同步。

### 7.2 写入

获得写入权限必须同时满足：

- 当前用户是活动项目成员。
- 当前用户是该任务的 Task Owner。
- 任务未归档且当前工作周期允许写入。
- 当前客户端持有该工作区唯一有效写入租约。

Project Owner、Project Admin、父任务 Owner 都不能因为身份而绕过上述条件。

### 7.3 写入租约

“一个 Task Owner”并不能自动避免同一成员在两台设备或两个 Agent 会话中并发写入，因此系统使用独占写入租约：

- 一个 Workspace 最多一个活动写入租约。
- 租约绑定用户、设备、客户端会话和工作周期。
- 客户端定期续租；异常退出后租约在 TTL 后失效。
- 同一 Owner 在另一设备请求写入时只能只读连接，或人工确认接管。
- 接管前优先要求原客户端同步和释放。
- 强制接管属于破坏性操作，需要确认并保留旧客户端最后同步版本。

### 7.4 Owner 转移

转移前必须满足：

- 没有未同步的本地变更，或原 Owner 已明确放弃并保存恢复副本。
- 活动写入租约已释放。
- 服务端已生成转移前快照。

转移完成后，新 Owner 才能获取写入租约。管理员可以发起转移，但不能借此读取未上传的本地文件或替代 Owner 写入。

## 8. Agent 权限与确认

### 8.1 身份继承

- Agent 操作以发起用户的成员身份执行。
- Agent 会话记录当前项目、连接任务、客户端和用户。
- Agent 不作为 Task Owner，也不获得逻辑角色之外的独立权限。
- 若需要管理员能力，用户必须先在可信界面开启管理员模式，并向 Agent 会话签发短时管理能力；Agent 仍不能自行续期或开启。

### 8.2 确认级别

| 级别 | 行为 | 示例 |
|---|---|---|
| A：只读 | 直接执行 | 读取任务、搜索 Wiki、读取工作区 |
| B：当前任务内安全写入 | 直接执行并记录 | 修改当前 owned 任务的标题/描述/验收/截止日期；在当前持有租约的工作区内新增或补丁式编辑文本 |
| C：结构或外部影响写入 | 必须人工确认 | 创建子任务、修改状态、依赖、Owner、父子关系、发表评论、完成任务 |
| D：破坏性或管理操作 | 必须展示影响并人工确认 | 归档、批量修改、删除文件、强制接管租约、管理员模式跨 Owner 修改 |

“任务内容外的所有修改都需要人工确认”在接口中落实为：仅 A、B 可以自动执行；C、D 均需确认。

### 8.3 确认不能替代授权

用户确认一个操作不代表系统应该放行。执行顺序必须是：

1. 解析和规范化请求。
2. 计算影响集合和所需权限。
3. 若当前用户无权，直接拒绝。
4. 若有权但需要确认，生成不可变操作提案。
5. 用户确认同一提案摘要。
6. 重新检查版本与权限后执行。

如果目标版本变化，旧确认失效，必须重新展示影响。

## 9. 评论与通知权限

- Member 和管理员可以发表评论；Viewer 只读。
- 用户只能编辑或删除自己的评论；管理员可以隐藏违规评论但保留审计副本。
- 评论不授予任务修改或工作区写入权限。
- `@成员`、Owner 转移、阻塞、依赖改变和管理员操作可以产生通知。
- Agent 发布评论属于任务内容外写入，必须确认。

## 10. 审计要求

以下操作必须记录成功与失败尝试：

- 管理员模式开启、关闭、过期和能力签发。
- 成员加入、移除、权限变化。
- Task Owner 转移。
- 依赖、层级、状态、归档和恢复。
- 写入租约获取、拒绝、接管和释放。
- Agent 提案、确认、拒绝和执行。
- 永久删除与历史导出。

审计记录至少包含操作者、代理类型、项目、目标、请求 ID、时间、前后版本、结果和原因。审计记录对普通成员可提供业务友好视图，安全原始日志仅 Project Owner 可访问。

## 11. 授权伪代码

```text
authorize(operation, actor, project, targets, session):
    require actor is authenticated
    membership = require active membership(actor, project)

    if operation is read:
        return membership.permission >= viewer

    if membership.permission == viewer:
        deny

    impact = calculate_impact(operation, targets)
    admin = session.admin_mode_active
            and membership.permission in {owner, admin}

    if operation touches workspace:
        require actor == task.owner
        require valid_write_lease(actor, session.device, workspace)

    if not admin:
        require normal_scope_allows(actor, operation, impact)

    if operation is from agent:
        require agent_confirmation_policy_satisfied(operation, impact)

    require domain_invariants_hold(operation, impact)
    allow
```
