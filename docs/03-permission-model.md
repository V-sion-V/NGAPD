# 权限与安全模型

文档状态：设计基线 0.4

相关文档：[领域模型](02-domain-model.md) · [Agent 设计](06-agent-integration.md) · [工作区设计](05-workspace-context-wiki.md)

## 1. 权限模型总览

系统存在三个必须分离的权限层：

1. **Web/任务权限**：决定用户能否修改项目数据、任务内容和任务结构。
2. **工作区权限**：分别决定用户能否读取或写入用户级、项目级、任务级文件工作区。
3. **Agent 权限与确认**：Agent 先继承当前用户权限，再根据操作风险要求人工确认。

任何后一层都不能扩大前一层授予的权限，但并非每类操作都同时使用三个授权层。最终条件按目标类型组合：

```text
task_data_allow = authenticated
    AND active_project_membership
    AND web_task_authorized

workspace_read_allow = authenticated
    AND workspace_read_policy_allows

workspace_write_allow = authenticated
    AND workspace_write_policy_allows
    AND valid_exclusive_write_lease

agent_allow = corresponding_underlying_allow
    AND agent_confirmation_satisfied_if_needed
```

用户级工作区不属于任何单一项目，因此不要求项目成员资格；其底层读取条件是“已认证”，写入条件是“对应用户本人 + 有效写入租约”。但 Agent 不得把底层可读解释为可自行发现或读取其他用户工作区：没有用户明确指定目标和目的时必须拒绝。项目级和任务级工作区仍要求目标项目的活动成员资格。

管理员模式主要扩大第一层任务权限。Project Owner/Admin 对项目级工作区的基础写入资格来自第二层，人通过 Web 或本地客户端写入不要求管理员模式；Agent 写入项目级工作区则额外要求用户明确请求并确认进入管理员模式。Project Owner 或 Project Admin 即使开启管理员模式，也不能写入其他 Task Owner 的任务级工作区。

## 2. 角色与能力分离

### 2.1 系统权限角色

| 权限角色 | 说明 |
|---|---|
| Project Owner | 每项目唯一，拥有项目生命周期、成员审批、管理员任命和项目级工作区写权限 |
| Project Admin | 有资格开启管理员模式，并拥有项目级工作区基础写入资格；默认任务操作仍按普通成员权限 |
| Member | 可以创建任务、管理自己的任务与任务级工作区，读取项目级工作区 |
| Viewer | 只读项目、任务、允许读取的工作区快照和 Wiki |

以上是项目内权限角色。无论用户在某项目中是何角色，都不影响其完整操作自己的用户级工作区，也不会扩大其对其他用户级工作区的只读权限。

### 2.2 逻辑角色

“程序员 L1”“角色原画 L2”等逻辑角色只描述能力。授权服务不得根据逻辑角色名称、等级、提示词或标签放行操作。

这意味着“程序员 L3”不会自动获得管理员权限，“项目管理员”也不会自动被视为某个专业角色。

## 3. 管理员模式

### 3.1 设计目的

Project Admin 的资格和管理员模式的启用状态分开，防止管理员在日常工作时误改其他成员的任务。

### 3.2 启用规则

- 只有 Project Owner 和活动状态的 Project Admin 可以启用。
- 必须由用户在可信界面中主动开启；Agent 可以展示进入管理员模式的提案，但只有用户显式确认后才能进入，不能自动进入。
- 开启后显示持续、明显的视觉提示。
- 使用短期管理会话，建议 30 分钟无操作后失效。
- 高风险部署可以要求重新验证密码或二次认证。
- 开启、关闭和过期均写入安全审计。
- Agent 可以无需再次确认地退出管理员模式，并应在任务完成、会话结束或不再需要管理权限时自动退出。

### 3.3 能力范围

开启管理员模式后，可以：

- 修改任意任务内容和状态。
- 修改任意合法的同级依赖。
- 修改任意任务 Owner。
- 移动任务、归档顶层任务、不可恢复删除非顶层任务。
- 处理跨 Owner 影响的任务子树。
- 管理项目角色、成员权限和项目配置（仍受 Project Owner 专属操作限制）。
- 代表当前用户的 Agent 可以在取得项目级工作区租约后写入项目级工作区；该能力必须由用户明确请求本次管理员模式后才生效。

管理员模式不能：

- 写入其他 Task Owner 的任务级工作区。
- 静默覆盖正在同步的工作区。
- 跳过 Agent 人工确认。
- 写入 `.ngapd/` 控制元数据。
- 修改不可变 Task Key 或历史审计。
- 代替 Project Owner 永久删除项目或任命新的 Project Owner。

## 4. 普通成员的任务权限

### 4.1 基本规则

普通模式下，成员只管理自己拥有的任务，以及自己有权管理的未分配子任务。父任务 Owner 不自动获得已分配给他人的后代任务内容权限。

“可以管理自己的节点及其子节点”采用以下保守含义：

- 可以在自己的任务下创建子任务。
- 可以继续管理尚未分配、且最近已分配祖先由自己拥有的子任务。
- 子任务一旦指派给其他成员，其内容、状态和任务级工作区即属于新 Owner 的权限范围。
- 父任务 Owner 可以读取后代状态和摘要，但不能修改其他 Owner 的节点。
- 依赖变更采用单独的双端责任规则；任何会改变其他 Owner 任务的 Owner、内容、状态、层级或存续状态的其他操作都需要管理员模式。

### 4.2 任务权限矩阵

| 操作 | 普通 Member | 管理员模式 | Viewer |
|---|---|---|---|
| 查看项目任务 | 允许 | 允许 | 允许 |
| 创建顶级任务 | 允许自我指派；指定他人时接受后才正式创建 | 允许，可未分配；指定他人仍需接受 | 禁止 |
| 在自己的任务下创建子任务 | 允许 | 允许 | 禁止 |
| 在他人任务下创建子任务 | 禁止 | 允许 | 禁止 |
| 修改自己任务的标题、正文、UTC 截止时间、逻辑角色、标签或展示类型 | 允许 | 允许 | 禁止 |
| 修改他人任务的标题、正文、UTC 截止时间、逻辑角色、标签或展示类型 | 禁止 | 允许 | 禁止 |
| 修改自己任务状态 | 允许，仍受状态机约束 | 允许，仍受状态机约束 | 禁止 |
| 修改他人任务状态 | 禁止 | 允许 | 禁止 |
| 把自己的任务转交给他人 | 允许发起；对方接受后才转交并失去写权限 | 允许发起；对方仍需接受 | 禁止 |
| 取得他人任务 Owner | 禁止 | 允许 | 禁止 |
| 移动自己的任务 | 仅当源、目标和影响集合均不涉及他人任务 | 允许 | 禁止 |
| 归档自己拥有的顶层任务 | 仅当影响集合均在自己权限内 | 允许 | 禁止 |
| 归档含他人任务的顶层子树 | 禁止 | 允许并展示影响预览 | 禁止 |
| 删除自己拥有的非顶层任务 | 仅当整个子树和依赖影响均在自己权限内；不可恢复 | 允许 | 禁止 |
| 删除含他人任务的非顶层子树 | 禁止 | 允许并展示不可恢复影响 | 禁止 |
| 恢复任务 | 不支持 | 不支持 | 不支持 |

### 4.3 未分配任务

- 未分配任务不能进入进行中或已完成。
- 自己拥有的任务下，未分配子任务由该最近祖先 Owner 暂时管理。
- 未分配任务指派给另一成员时，目标成员接受后管理权才转移；自我指派可以直接完成。
- 为避免产生无管理者的顶级节点，普通成员创建顶级任务时必须指定 Owner。
- 管理员模式可以创建和整理顶级未分配任务。
- 成员被移除时，其拥有的任务统一转为未分配；未分配子任务由最近的已分配祖先 Owner 暂时管理，顶层未分配任务只能由管理员模式重新指派。

`created_by` 仅用于审计，不把创建者变成第二 Task Owner。

## 5. 依赖权限

依赖 `predecessor -> successor` 连接两个同级启用任务。不开启管理员模式时，当前用户拥有两个端点，或拥有两者共同的直接父任务，可以直接添加或移除依赖。只拥有一个端点时只能发起依赖变更请求，另一端点 Owner 接受后才生效。顶层任务由 Project Owner 作为虚拟根作用域控制者直接处理。其他情况必须进入管理员模式。

### 5.1 普通模式允许

- 当前用户同时拥有 predecessor 和 successor，可以直接添加或移除合法依赖。
- 当前用户拥有两端共同的直接父任务时，即使两个子任务都由他人拥有，也可以修改两者之间的依赖。
- 顶层任务统一属于虚拟项目根节点；Project Owner 视为该根作用域控制者，可以在普通模式下修改顶层任务依赖。Project Admin 若未开启管理员模式且不拥有任一端点，不获得该根作用域控制权。
- 当前用户只拥有一个端点时，可以提交绑定当前 `graph_version` 的变更请求；另一端点 Owner 接受且图版本未变化后执行。
- 未分配端点由其最近已分配祖先 Owner 代表接受；顶层未分配端点没有代表时只能由 Project Owner 或管理员模式处理。
- 两端必须处于启用态、属于同一父级作用域，操作后仍必须无环。

### 5.2 普通模式禁止

- 在两个都由他人拥有、且共同直接父任务也不由当前用户拥有的任务之间直接添加或移除依赖。
- 只拥有一个端点时绕过另一端点 Owner 的接受直接执行。
- 对两个都未分配、且不在自己临时管理范围内的任务建立依赖。
- 使用批量操作夹带修改当前用户既不拥有任一端点、也不拥有共同直接父任务的其他依赖。

上述直接操作需要管理员模式。依赖接口必须同时解析两个端点、两端 Owner、共同直接父级和 `graph_version`，不能只检查当前页面或当前选中任务。

### 5.3 与移动、归档的区别

依赖规则仅授权显式的依赖增删或发起变更请求，不自动授权移动、归档或删除任务。后者可能批量改变子树和多个外部依赖，仍按完整影响集合校验；普通成员不能借删除自己拥有的端点绕过对其他任务的保护。

## 6. 影响集合授权

移动、顶层归档、非顶层不可恢复删除和 Owner 转移不能只检查目标任务。领域服务必须先计算影响集合：

- 目标任务及其全部后代。
- 将被删除或失效的依赖边。
- 因依赖变化而改变有效状态的后继任务。
- 已完成祖先的完成合法性。
- 活动工作区写入租约和未同步版本。
- 对应父级作用域的 `graph_version`。

普通模式要求影响集合内所有需要写入的已分配任务都由当前用户拥有；否则拒绝并列出需要管理员处理的原因。

## 7. 工作区权限

### 7.1 读取

- 用户级工作区：对应用户可读；其他已认证用户只读。匿名用户和未认证连接不可读。
- 项目级工作区：活动项目成员和 Viewer 可读；项目外用户不可读。
- 任务级工作区：活动项目成员和 Viewer 可读；项目外用户不可读。
- 已被移除的项目成员立即失去该项目的项目级与任务级工作区读取权限，但仍可按用户级规则读取用户工作区。
- 本地已经下载的副本无法通过服务端远程擦除，客户端应提示其权限已失效并停止同步。
- Agent 读取其他用户的用户级工作区还必须具有本次会话中的明确用户指令；不能通过枚举、搜索或上下文发现自行扩大读取范围。

### 7.2 写入

写入者资格按作用域决定：

| 工作区类型 | 可申请写入租约的用户 |
|---|---|
| 用户级 | 对应用户本人 |
| 项目级 | 该项目的 Project Owner 或 Project Admin；人直接写入无需管理员模式，Agent 写入还要求显式管理员模式 |
| 任务级 | 当前 Task Owner |

真正获得写入权限还必须同时满足：

- 当前用户仍符合对应作用域的写入者资格。
- 工作区处于允许写入的生命周期状态；已完成任务的任务级工作区除外，必须先重新打开任务。
- 当前客户端持有该工作区唯一有效写入租约。

任何角色都不能绕过写入租约。Project Owner、Project Admin、父任务 Owner 不能因为身份写入他人的任务级工作区；Task Owner 也不能因此写入项目级或其他用户的用户级工作区。`.ngapd/` 控制元数据不属于可同步用户内容，任何 Agent 都不得写入。

### 7.3 写入租约

固定的写入者集合不能避免同一用户多设备，或多个 Project Owner/Admin 争用同一项目级工作区，因此三种工作区都使用独占写入租约：

- 一个 Workspace 最多一个活动写入租约。
- 租约绑定用户、设备、客户端会话和工作周期。
- 客户端定期续租；异常退出后租约在 TTL 后失效。
- 另一名合格写入者或同一用户的另一设备请求写入时只能只读连接，或人工确认接管。
- 接管前优先要求原客户端同步和释放。
- 强制接管属于破坏性操作，需要确认并保留旧客户端最后同步版本。
- 租约失效后产生或残留的本地未同步改动不具有提交资格；下次连接直接使用服务端权威清单替换发生改动的受管文件，不合并、不导入、不创建可提交分支。

### 7.4 Owner 转移

本节只适用于任务级工作区的 Task Owner 转移。转移前必须满足：

- 服务端已接收当前有效租约下的最终同步；无效租约下的本地变更一律不作数。
- 活动写入租约已释放。
- 服务端已生成转移前快照。
- 目标成员已明确接受 Owner 转移。

转移完成后，新 Task Owner 才能获取任务级写入租约。管理员可以发起转移，但目标成员仍需接受；成员移除导致 Owner 置空不需要接受。任何已失效租约下的本地文件都不能上传。

项目级工作区的写入者随 Project Owner/Admin 资格变化：资格被撤销时立即停止续租；若该用户持有租约，服务端撤销租约并保留最后同步版本。用户级工作区的写入者固定为对应用户。

## 8. Agent 权限与确认

### 8.1 身份继承

- Agent 操作以发起用户的成员身份执行。
- Agent 会话记录当前项目、连接任务、客户端和用户。
- Agent 不作为任何工作区的独立所有者，只继承当前用户在目标工作区上的读写资格。
- 若需要管理员能力，Agent 必须生成“进入管理员模式”提案；只有用户显式确认后，可信界面才向该 Agent 会话签发短时管理能力。Agent 不能自动进入或自行续期，但可以自动退出并销毁能力。

### 8.2 确认级别

| 级别 | 行为 | 示例 |
|---|---|---|
| A：任务与知识读取 | 直接执行 | 读取全部有权查看的任务信息、角色、评论、Wiki 和只读工作区；其他用户的用户级工作区还要求明确用户指令 |
| B：用户级/任务级可写工作区完整访问 | 直接执行并记录 | 在持有租约的本人用户级或本人拥有的任务级工作区内创建、修改、移动、重命名和删除用户内容文件，并执行正常同步 |
| B+：项目级工作区完整访问 | 显式管理员模式后直接执行并记录 | 用户明确要求 Agent 进入管理员模式，确认成功且当前用户是 Project Owner/Admin，并持有项目级工作区租约 |
| C：任务管理数据修改 | 必须人工确认 | 修改标题或统一正文、UTC 截止时间、逻辑角色、展示类型、创建子任务、修改状态、依赖、Owner、父子关系、发表评论、完成任务 |
| D：管理员或系统级操作 | 必须展示影响并人工确认 | 进入管理员模式、跨 Owner 修改、顶层归档、非顶层不可恢复删除、批量修改、强制接管租约、项目永久删除 |

Agent 权限在接口中落实为：任务管理信息读取无需确认；读取其他用户的用户级工作区必须来自明确用户指令；本人用户级和本人任务级工作区在持有租约时可完整访问用户内容；项目级工作区写入还要求用户明确请求并确认管理员模式。任何任务管理数据修改都属于 C 或 D，必须人工确认。工作区完整访问权不等于可以写 `.ngapd/`、绕过租约、访问工作区外路径或改写服务端任务数据。

### 8.3 确认不能替代授权

用户确认一个操作不代表系统应该放行。执行顺序必须是：

1. 解析和规范化请求。
2. 计算影响集合和所需权限。
3. 若当前用户无权，直接拒绝。
4. 若有权但需要确认，生成不可变操作提案。
5. 用户确认同一提案摘要。
6. 重新检查版本与权限后执行。

如果目标 Task 版本、所属 `graph_version`、权限或影响集合指纹变化，旧确认失效，必须重新展示影响。

## 9. 评论与通知权限

- Member 和管理员可以发表评论；Viewer 只读。
- 用户只能编辑或删除自己的评论；管理员可以隐藏违规评论但保留审计副本。
- 评论不授予任务修改或工作区写入权限。
- `@成员`、Owner 转移、阻塞、依赖改变和管理员操作可以产生通知。
- Agent 发布评论属于任务管理数据修改，必须确认。

## 10. 审计要求

以下操作必须记录成功与失败尝试：

- 管理员模式开启、关闭、过期和能力签发。
- 成员加入、移除、权限变化。
- Task Owner 转移。
- 依赖请求/接受、图版本、层级、状态、顶层归档和非顶层不可恢复删除。
- 写入租约获取、拒绝、接管和释放。
- Agent 提案、确认、拒绝和执行。
- `[M7]` 项目永久删除与历史导出。

审计记录至少包含操作者、代理类型、项目、目标、请求 ID、时间、前后版本、结果和原因。审计记录对普通成员可提供业务友好视图，安全原始日志仅 Project Owner 可访问。

## 11. 授权伪代码

```text
authorize(operation, actor, project, targets, session):
    require actor is authenticated

    if operation accesses user workspace:
        if operation is read:
            if operation from agent and workspace does not belong to actor:
                require explicit_user_instruction_for_this_workspace(operation)
            allow
        require workspace belongs to actor
        require valid_write_lease(actor, session.device, workspace)
        require target_path not under ".ngapd/"
        require target_path not in {"TASK.md", "SUMMARY.md"}
        require domain_invariants_hold(operation, workspace)
        allow

    membership = require active membership(actor, project)

    if operation reads project or task workspace:
        allow

    if operation is normal project or task workspace file write:
        require workspace_write_policy_allows(actor, workspace.scope)
        require valid_write_lease(actor, session.device, workspace)
        require target_path not under ".ngapd/"
        require target_path not in {"TASK.md", "SUMMARY.md"}
        if operation from agent and workspace.scope == project:
            require session.admin_mode_active
            require session.admin_mode_entered_from_explicit_user_request
        require domain_invariants_hold(operation, workspace)
        allow

    if operation is task_or_knowledge_read:
        return membership.permission >= viewer

    if membership.permission == viewer:
        deny

    impact = calculate_impact(operation, targets)
    admin = session.admin_mode_active
            and membership.permission in {owner, admin}

    if not admin:
        require normal_scope_allows(actor, operation, impact)

    if operation is task_management_write from agent:
        require agent_confirmation_policy_satisfied(operation, impact)

    require domain_invariants_hold(operation, impact)
    allow
```
