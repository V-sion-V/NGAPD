# 实施路线与验收策略

文档状态：设计基线 0.4

相关文档：[产品需求](01-product-requirements.md) · [系统架构](04-system-architecture.md) · [决策记录](08-decisions-and-open-issues.md)

## 1. 实施原则

- 先验证递归任务规则、工作区和 Agent 三个差异化能力，再扩充通用项目管理功能。
- 先固化领域约束，再开发多个写入入口。
- Web UI、本地客户端和 Agent 不各自实现一套权限逻辑。
- 每个里程碑必须产生可端到端演示的结果。
- 不以预计团队规模作为正确性限制，但围绕实际规模优化体验。

## 2. 前置技术验证

在完成大量业务页面前，先开展三个有限原型。

本节保留产品侧验证对象，但具体设备矩阵、P95 延迟、数据规模、同步吞吐和恢复时间阈值由另立的非功能需求文档冻结；在该文档完成前，“流畅”和“性能基线”只表示待量化的验收项。

### 2.1 平铺树状任务界面原型

目标：验证不依赖语义缩放的层级导航、同级依赖和任务详情体验。

数据集：

- 5—8 层任务深度。
- 一个父任务下 200 个直接子任务。
- 顶层任务之间的虚拟根节点依赖。
- 普通、冲刺、里程碑三种仅影响外观的任务。
- 稀疏、中等和较密集的无环依赖。

验证点：

- 本级任务列表、当前任务详情和直接子任务列表。
- 层级面包屑和返回上一级按钮。
- 同级依赖展示和顶层任务依赖。
- 搜索定位、过滤和层级恢复。
- 三种展示类型是否仅改变外观、不改变交互。

退出条件：可以在目标设备上流畅浏览 200 个同级任务，并通过面包屑、子任务列表和返回按钮导航任意深度。

### 2.2 工作区租约与同步原型

目标：验证 macOS/Windows 文件差异，以及用户级、项目级、任务级工作区的统一单写入语义。

验证点：

- 首次物化、文本修改、图片上传、删除和重命名。
- 同一 Owner 两台设备争用租约。
- 两名 Project Admin 争用同一个项目级工作区租约。
- 用户级、项目级、任务级写入资格和只读范围。
- 客户端崩溃、租约过期、恢复和强制接管。
- Owner 转移前存在未同步文件。
- Windows/macOS 文件名、大小写和路径差异。
- 租约失效或服务端版本不匹配后，本地改动不上传，并在下次连接时由服务端权威版本替换。
- `.ngapd/`、`TASK.md` 和 `SUMMARY.md` 不进入用户内容同步清单。

退出条件：任一时刻最多一个服务端认可的写入者；服务端最后已同步版本始终权威；失效租约下的本地改动不会上传，并能在重新连接时被服务端版本确定性替换。

### 2.3 Agent 上下文原型

目标：验证任务边界能否产生高质量、低污染上下文。

数据集：

- 用户级个人流程/Skill、项目级通用规则/Skill、成员多角色提示。
- 多层祖先任务。
- 多个已完成 predecessor 摘要。
- 当前工作区中长短不一的过程文档和图片引用。

验证点：

- Agent 是否能读取当前用户的全部逻辑角色描述，并正确理解当前任务统一正文中的目标、说明和验收信息。
- 项目规则与用户个人流程冲突时，是否遵循既定优先级。
- 其他用户的用户级 Skill 是否不会被自动加载。
- 没有明确用户指令时，Agent 是否不会发现、搜索或读取其他用户的用户级工作区。
- 没有用户明确要求并确认管理员模式时，Agent 是否不能写入项目级工作区。
- 是否会错误使用无关兄弟任务内容。
- 上下文超预算时是否保留关键规则。
- 工作区中的提示注入文本是否会改变权限行为。
- 摘要能否帮助另一名成员继续工作。

退出条件：Agent 可通过清单渐进读取完成任务分析，工具授权不依赖模型是否正确理解文件内容。

## 3. 开发里程碑

### M0：领域基线和工程骨架

交付：

- 项目模块化边界和迁移框架。
- Project Key、Task Key 分配。
- 任务树、同级 DAG、状态机和影响集合的领域测试。
- 每父级 `graph_version`、跨 Owner 依赖接受和并发失效测试。
- 统一错误码、审计接口和 API 版本约定。
- Web、服务端和本地客户端的最小构建/发布流水线。

验收门槛：不通过 UI 也能用领域/API 测试证明树、依赖和状态不变量。

### M1：项目、角色和成员

交付：

- 注册登录、项目创建/归档/恢复。
- 用户注册时创建用户级逻辑工作区，项目创建时创建项目级逻辑工作区。
- 加入申请和审批。
- 成员移除时清空其 Task Owner；Project Owner 转移由接收者接受且项目始终有唯一 Owner。
- Project Owner/Admin/Member/Viewer。
- 管理员模式开关和审计。
- 系统角色模板、项目快照、多角色绑定和自我介绍。

验收门槛：管理员资格不等于自动启用管理员模式；逻辑角色不能授予权限。

### M2：任务管理闭环

交付：

- 多层任务树和任务详情。
- 单 Task Owner、接收者接受跨成员指派、统一任务正文、可选任务逻辑角色、UTC 截止时间、标签和仅影响外观的展示类型。
- 同级依赖、`graph_version`、跨 Owner 变更接受与环检测。
- 基础/有效状态、人工阻塞、完成和重新打开。
- 顶层归档、非顶层不可恢复删除、Owner 转移和活动流。
- 父任务满足完成条件时的“可完成”投影与 Owner 通知。
- 普通模式与管理员模式权限。
- 评论和关键通知。

验收门槛：跨 Owner 影响操作不能通过低层 API 绕过；父任务不会因子任务完成而自动完成。

### M3：平铺树状任务界面

交付：

- 虚拟项目根节点下的顶层任务列表和顶层依赖。
- 本级任务列表、当前任务详情和直接子任务列表。
- 层级面包屑和返回上一级按钮。
- 普通、冲刺、里程碑的差异化外观。
- 搜索定位、过滤和单层 200 任务性能验证。

验收门槛：依赖方向和层级边界始终正确；用户能从项目顶层定位到任意深层 Task Key，并能稳定返回上一级。

### M4：工作区客户端

交付：

- macOS、Windows 客户端基础安装和登录。
- 统一 NGAPD 根目录配置，以及用户级、项目级、任务级稳定路径和安全校验。
- 三种逻辑工作区的物化。
- 用户级本人写、项目级 Project Owner/Admin 写、任务级 Task Owner 写的权限策略。
- 多只读连接、单写连接、独占租约和续租。
- 文本/图片清单同步、版本和服务端快照。
- `.ngapd/` 控制面与用户内容清单分离。
- 服务端生命周期、本地副本状态和连接/租约状态分离。
- Owner 转移、冻结，以及租约失效后由服务端版本替换本地改动。

验收门槛：用户只能写自己的用户级工作区；Project Owner/Admin 可以写项目级工作区但不能写他人的任务级工作区；Task Owner 只能写自己的任务级工作区；任一工作区的第二个连接都无法产生第二个合法写入流。

### M5：Agent 与 Skill

交付：

- 标准工具协议服务。
- NGAPD Skill。
- `connect_task` 与通用 `connect_workspace`，以及三级工作区读写和同步。
- 用户级个人流程/Skill、项目级通用规则/Skill、任务级过程文件的分层上下文。
- 全部授权任务信息的无确认读取。
- 其他用户的用户级工作区只有在用户明确指定目标和目的后才允许读取。
- 对当前用户有写资格且持有租约的用户级或任务级工作区进行完整用户内容访问，包括创建、修改、移动、重命名和删除。
- Agent 写项目级工作区必须由用户明确要求并确认进入管理员模式。
- 对标题、统一正文等任何任务管理数据修改的提案确认。
- 创建/拆分、状态、带图版本的依赖及接受、Owner、逻辑角色、展示类型、完成、顶层归档和非顶层删除提案。
- 显式进入、自动退出管理员模式。
- 人工确认、版本失效和 Agent 审计。

验收门槛：Agent 无法自行确认或自动进入管理员模式，也不能通过编辑本地机器文件绕过领域服务；退出管理员模式可以自动完成。

### M6：完成摘要和 Wiki

交付：

- 完成快照。
- 摘要草稿、人工修订和确认。
- 摘要作为独立 KnowledgeEntry 管理，`SUMMARY.md` 只作为只读投影。
- 按任务树浏览的 Wiki。
- 全文搜索、来源链接和摘要版本历史。
- 摘要失败重试和索引重建。

验收门槛：知识条目能追溯到具体 Task Key、工作周期、文件和同步版本。

### M7：额外稳定性与发布增强（MVP 后）

交付：

- 备份恢复、回收站和保留策略。
- 增强安全审计、速率限制和异常告警。
- 长期容量调优、客户端自动更新策略和扩展部署文档。
- 示例项目、系统角色模板和新用户引导。
- 开源许可证、贡献指南和安全问题报告流程。

验收门槛：额外能力启用后，在干净 Linux 环境验证相应部署、恢复、安全和更新流程。本里程碑不属于 M0—M6 MVP 的发布质量门槛。

## 4. MVP 截止范围

MVP 截止在 M6 的基础能力。M7 的备份恢复、回收站、增强安全告警、自动更新策略和开源项目治理属于额外需求，不纳入 MVP 发布门槛。以下功能同样进入后续版本：

- 语义缩放、节点内嵌子图、无限画布和高级动画。
- 邮件、聊天软件等外部通知渠道。
- Git 提交、分支、PR 的深度集成。
- 向量搜索和跨项目知识推荐。
- 自定义任务类型、字段和状态机。
- 多个连接共同写入同一个工作区。
- 移动端原生客户端。

## 5. 测试策略

### 5.1 领域属性测试

使用随机生成任务树和依赖图验证：

- 任意合法操作后父子关系无环。
- 任意合法依赖操作后同级图无环。
- 不可能生成跨父依赖。
- 任务编号不复用。
- 每个用户、项目、任务成功创建后分别恰有一个对应类型逻辑工作区，且重复创建被唯一约束拒绝。
- 父任务完成时全部启用态直接子任务完成。
- 父任务首次满足完成条件时只产生一次“可完成”提示，不自动完成。
- 有活动阻塞的未完成任务有效状态必为 blocked。
- 删除、移动和转交操作的影响集合不遗漏外部 successor。

### 5.2 权限矩阵测试

至少覆盖：

- Project Owner/Admin/Member/Viewer。
- 管理员模式关闭、开启、过期。
- 自己、他人、未分配任务。
- 自己节点下由他人拥有的后代。
- 修改依赖时自己拥有两端、拥有共同直接父任务、只拥有一个端点并等待另一端接受、两端都不拥有等情况。
- 归档包含他人任务的顶层子树、删除包含他人任务或影响他人 successor 的非顶层子树。
- 管理员试图写其他 Task Owner 的任务级工作区。
- 人直接写项目级工作区、Project Admin 的 Agent 未开启管理员模式时写项目级工作区，以及写他人任务级工作区的对照测试。
- 用户本人和其他已认证用户对用户级工作区的底层读写差异，以及 Agent 缺少明确用户指令时不得跨用户读取。
- Agent 已确认但用户本身无权的操作。

所有公开写入入口都运行同一组授权契约测试。

### 5.3 工作区测试

- 清单和内容哈希正确性。
- 临时文件原子替换。
- 租约续租、过期、接管和时钟偏差。
- Owner 转移与同步事务。
- 新建/未分配/转移场景的 Task Owner 接受、成员移除后 Task Owner 批量置空，以及 Project Owner 接受式转移。
- 两名合格 Project Admin 对同一项目级工作区争用唯一租约。
- 三种工作区各自的读取者、写入者资格变化和租约撤销。
- macOS/Windows 路径、大小写、保留名称和 Unicode。
- 符号链接及路径穿越。
- 文本、图片、超限文件和排除规则。
- `.ngapd/` 和服务端投影文件不会进入同步清单。
- 租约失效或版本不匹配后，本地冲突文件被服务端版本替换。

### 5.4 Agent 测试

- 每个工具的允许、拒绝、确认和版本过期路径。
- Agent 修改标题、统一正文、UTC 截止时间、逻辑角色、标签或展示类型也必须获得人工确认。
- Agent 可以无需任务管理确认地完整操作当前用户有写资格且持有租约的用户级或任务级用户内容；项目级还要求显式管理员模式；任何作用域都不能访问保护路径、工作区外路径或绕过租约。
- Agent 不会在缺少明确用户指令时发现或读取其他用户的用户级工作区。
- Agent 自动发现当前用户自用 Skill 和当前项目通用 Skill，不自动加载其他用户 Skill。
- Agent 不能自动确认自己的提案。
- Agent 不能自动进入管理员模式，可以在规定场景自动退出。
- 文件中的提示注入不影响工具权限。
- 完成前必须取得服务端最终同步版本；存在有效租约但无法完成同步、存在未完成启用子任务或 blocker 时拒绝。失效租约下的本地改动不进入完成快照。
- 重试不会重复创建任务或依赖；`graph_version` 变化后旧依赖确认失效。

### 5.5 初版任务界面测试

- 200 同层节点和不同密度依赖。
- 项目根顶层依赖、本级列表、任务详情、子任务列表、面包屑、返回上一级和搜索跳转。
- 长标题、多语言、缺失 Owner 和超期状态。
- 普通、冲刺、里程碑在颜色之外也有可区分标识。
- 键盘导航、列表虚拟化和依赖方向的可访问性表达。

## 6. 端到端验收场景

### 场景 A：跨职业成员

一个成员同时绑定“玩法程序员 L2”和“技术美术 L1”，创建任务后 Agent 能读取两个角色的能力边界，但成员不会因此获得管理员权限。

### 场景 B：父子任务责任隔离

A 拥有父任务并创建子任务后转交给 B。A 可以查看 B 的进展，但不能修改 B 的任务内容、状态或任务级工作区；管理员模式可以修改 Web 任务结构，但仍不能写 B 的任务级工作区。

### 场景 C：依赖授权与接受

A 同时拥有 predecessor 和 successor，或拥有两者共同的直接父任务时，可以直接添加或移除合法同级依赖。A 只拥有一个端点时先发起请求，由另一端点 Owner 接受；`graph_version` 变化后请求失效。顶层任务由 Project Owner 作为虚拟根控制者处理；其他情况必须进入管理员模式。新增边产生环时一律拒绝。

### 场景 D：三级工作区单写入

A 在 macOS 获得用户级或任务级写入租约后，Windows 客户端只能只读；两名 Project Admin 连接项目级工作区时也只有一人能获得写入租约。人工确认接管后，旧租约失效，旧连接不能再提交；旧连接的本地改动在重新连接时由服务端版本替换。

### 场景 E：分层规则与 Skill

项目级工作区包含团队编码规则和通用 Skill，用户级工作区包含个人检查流程和自用 Skill。Agent 连接任务后可以读取两层内容，只自动加载当前用户的自用 Skill；没有明确用户指令时不读取其他用户工作区。Agent 修改项目规则前必须由用户明确要求并确认进入管理员模式。项目规则与个人流程冲突时项目规则优先。

### 场景 F：Agent 拆分和完成

Agent 完整操作当前有写权限且持有租约的用户级或任务级用户内容无需任务管理确认，但拆分子任务和修改任何任务管理字段必须展示提案。子任务未完成时完成提案被拒绝；全部启用子任务完成后父任务 Owner 收到“可完成”提示，用户确认完成后系统使用服务端最终同步版本创建快照、冻结任务级工作区，并生成独立摘要草稿。

## 7. 发布质量门槛

- 所有领域不变量有自动化测试。
- 三层权限矩阵有 API 级契约测试。
- 任一客户端崩溃场景不会覆盖最后服务端同步版本。
- 所有 Agent 任务管理写入都可追溯到人工确认或明确拒绝。
- 平铺树状任务界面在目标规模下满足另立非功能需求文档中的交互性能基线。
- 用户文档明确区分“本地已保存”“已同步”“已完成并快照”。

## 8. 后续评估指标

不考虑商业指标，开源自用阶段建议关注产品有效性：

- 从连接任务到获得可用上下文所需操作数。
- Agent 首次读取后正确复述用户逻辑角色范围和任务统一正文要点的比例。
- 完成任务中包含过程文档和确认摘要的比例。
- 新成员通过 Wiki 理解已完成任务所需的追问次数。
- 被权限层阻止的真实越权操作数与误拦截数。
- 按用户级、项目级、任务级分别统计未同步、租约争用、接管和恢复事件数。
- 平铺树状界面中通过搜索、面包屑和返回上级定位任务的成功率。
