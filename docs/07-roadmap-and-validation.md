# 实施路线与验收策略

文档状态：设计基线 0.1  
相关文档：[产品需求](01-product-requirements.md) · [系统架构](04-system-architecture.md) · [决策记录](08-decisions-and-open-issues.md)

## 1. 实施原则

- 先验证任务图、工作区和 Agent 三个差异化能力，再扩充通用项目管理功能。
- 先固化领域约束，再开发多个写入入口。
- Web UI、本地客户端和 Agent 不各自实现一套权限逻辑。
- 每个里程碑必须产生可端到端演示的结果。
- 不以预计团队规模作为正确性限制，但围绕实际规模优化体验。

## 2. 前置技术验证

在完成大量业务页面前，先开展三个有限原型。

### 2.1 递归语义缩放原型

目标：验证嵌套节点、局部 DAG 和布局性能。

数据集：

- 5—8 层任务深度。
- 一个父任务下 200 个直接子任务。
- 多个折叠和展开节点。
- 稀疏、中等和较密集的无环依赖。

验证点：

- 远景、中景、近景档位切换。
- 聚焦节点和面包屑。
- 局部布局是否稳定，展开子图是否导致整个项目跳动。
- 搜索定位、过滤、选择和详情面板。
- 布局计算是否可以移出主线程。

退出条件：可以在目标设备上流畅选择和导航，且不需要一次渲染整个任务树。

### 2.2 工作区租约与同步原型

目标：验证 macOS/Windows 文件差异和单写入语义。

验证点：

- 首次物化、文本修改、图片上传、删除和重命名。
- 同一 Owner 两台设备争用租约。
- 客户端崩溃、租约过期、恢复和强制接管。
- Owner 转移前存在未同步文件。
- Windows/macOS 文件名、大小写和路径差异。
- 服务端版本不匹配时不会静默覆盖。

退出条件：任一时刻最多一个服务端认可的写入者；异常场景均能保留最后已同步版本，并明确展示本地未同步风险。

### 2.3 Agent 上下文原型

目标：验证任务边界能否产生高质量、低污染上下文。

数据集：

- 项目规范、成员多角色提示。
- 多层祖先任务。
- 多个已完成 predecessor 摘要。
- 当前工作区中长短不一的过程文档和图片引用。

验证点：

- Agent 是否能正确理解任务目标和验收条件。
- 是否会错误使用无关兄弟任务内容。
- 上下文超预算时是否保留关键规则。
- 工作区中的提示注入文本是否会改变权限行为。
- 摘要能否帮助另一名成员继续工作。

退出条件：Agent 可通过清单渐进读取完成任务分析，工具授权不依赖模型是否正确理解文件内容。

## 3. 开发里程碑

### M0：领域基线和工程骨架

交付：

- 项目模块化边界和迁移框架。
- Project Key、Task Key 分配。
- 任务树、同级 DAG、状态机和影响集合的领域测试。
- 统一错误码、审计接口和 API 版本约定。
- Web、服务端和本地客户端的最小构建/发布流水线。

验收门槛：不通过 UI 也能用领域/API 测试证明树、依赖和状态不变量。

### M1：项目、角色和成员

交付：

- 注册登录、项目创建/归档。
- 加入申请和审批。
- Project Owner/Admin/Member/Viewer。
- 管理员模式开关和审计。
- 系统角色模板、项目快照、多角色绑定和自我介绍。

验收门槛：管理员资格不等于自动启用管理员模式；逻辑角色不能授予权限。

### M2：任务管理闭环

交付：

- 多层任务树和任务详情。
- 单 Task Owner、截止日期、验收条件和标签。
- 同级依赖与环检测。
- 基础/有效状态、人工阻塞、完成和重新打开。
- 归档、恢复、Owner 转移和活动流。
- 普通模式与管理员模式权限。
- 评论和关键通知。

验收门槛：跨 Owner 影响操作不能通过低层 API 绕过；父任务不会因子任务完成而自动完成。

### M3：任务图与语义缩放

交付：

- 顶层任务图。
- 3—4 档离散语义缩放。
- 嵌套子图、局部布局、聚焦和面包屑。
- 搜索定位、过滤和详情面板。
- 单层 200 节点性能验证。

验收门槛：依赖方向和嵌套边界始终正确；用户能从项目顶层定位到任意深层 Task Key。

### M4：工作区客户端

交付：

- macOS、Windows 客户端基础安装和登录。
- 项目根目录配置和路径安全校验。
- 逻辑工作区物化。
- 读连接、Owner 写连接、独占租约和续租。
- 文本/图片清单同步、版本和服务端快照。
- Owner 转移、冻结和异常恢复。

验收门槛：其他成员、Project Admin 和 Project Owner 均无法写入非 owned 工作区；同一 Owner 的第二设备无法产生第二个合法写入流。

### M5：Agent 与 Skill

交付：

- 标准工具协议服务。
- NGAPD Skill。
- 连接、上下文、任务读取、工作区读写和同步。
- 当前任务内容安全更新。
- 创建/拆分、状态、依赖、Owner、完成和归档提案。
- 人工确认、版本失效和 Agent 审计。

验收门槛：Agent 无法自行确认、开启管理员模式或通过编辑本地机器文件绕过领域服务。

### M6：完成摘要和 Wiki

交付：

- 完成快照。
- 摘要草稿、人工修订和确认。
- 按任务树浏览的 Wiki。
- 全文搜索、来源链接和摘要版本历史。
- 摘要失败重试和索引重建。

验收门槛：知识条目能追溯到具体 Task Key、工作周期、文件和同步版本。

### M7：稳定性与发布

交付：

- 备份恢复、回收站和保留策略。
- 安全审计、速率限制和异常告警。
- 性能基线、客户端自动更新策略和部署文档。
- 示例项目、系统角色模板和新用户引导。
- 开源许可证、贡献指南和安全问题报告流程。

验收门槛：在干净 Linux 环境完成自托管部署，在 macOS/Windows 完成同一项目的只读/写入协作演示。

## 4. MVP 截止范围

MVP 建议截止在 M6 的基础能力。以下功能进入后续版本：

- 完全连续的无限画布和高级动画。
- 依赖变更请求工作流。
- 邮件、聊天软件等外部通知渠道。
- Git 提交、分支、PR 的深度集成。
- 向量搜索和跨项目知识推荐。
- 自定义任务类型、字段和状态机。
- 多人共同写入一个任务工作区。
- 移动端原生客户端。

## 5. 测试策略

### 5.1 领域属性测试

使用随机生成任务树和依赖图验证：

- 任意合法操作后父子关系无环。
- 任意合法依赖操作后同级图无环。
- 不可能生成跨父依赖。
- 任务编号不复用。
- 父任务完成时全部直接子任务完成。
- 有活动阻塞的未完成任务有效状态必为 blocked。
- 删除、移动和转交操作的影响集合不遗漏外部 successor。

### 5.2 权限矩阵测试

至少覆盖：

- Project Owner/Admin/Member/Viewer。
- 管理员模式关闭、开启、过期。
- 自己、他人、未分配任务。
- 自己节点下由他人拥有的后代。
- 自己 successor 引用他人 predecessor。
- 归档包含他人任务或影响他人 successor 的节点。
- 管理员试图写其他 Owner 工作区。
- Agent 已确认但用户本身无权的操作。

所有公开写入入口都运行同一组授权契约测试。

### 5.3 工作区测试

- 清单和内容哈希正确性。
- 临时文件原子替换。
- 租约续租、过期、接管和时钟偏差。
- Owner 转移与同步事务。
- macOS/Windows 路径、大小写、保留名称和 Unicode。
- 符号链接及路径穿越。
- 文本、图片、超限文件和排除规则。
- 服务端恢复后对象与清单一致。

### 5.4 Agent 测试

- 每个工具的允许、拒绝、确认和版本过期路径。
- Agent 不能夹带 Owner/状态字段到内容更新工具。
- Agent 不能自动确认自己的提案。
- 文件中的提示注入不影响工具权限。
- 完成前有未同步文件、未完成子任务或 blocker 时拒绝。
- 重试不会重复创建任务或依赖。

### 5.5 可视化测试

- 200 同层节点和不同密度依赖。
- 多层展开、折叠和搜索跳转。
- 长标题、多语言、缺失 Owner 和超期状态。
- 布局版本变化与动画中断。
- 键盘导航、缩放可访问性和颜色之外的状态表达。

## 6. 端到端验收场景

### 场景 A：跨职业成员

一个成员同时绑定“玩法程序员 L2”和“技术美术 L1”，创建任务后 Agent 能读取两个角色的能力边界，但成员不会因此获得管理员权限。

### 场景 B：父子任务责任隔离

A 拥有父任务并创建子任务后转交给 B。A 可以查看 B 的进展，但不能修改 B 的任务内容、状态或工作区；管理员模式可以修改 Web 任务结构，但仍不能写 B 的工作区。

### 场景 C：依赖授权

A 可以让自己的任务依赖 B 的同级任务；A 不能删除一个会改变 B 任务可执行性的依赖。新增边产生环时，无论管理员模式是否开启都必须拒绝。

### 场景 D：同 Owner 多设备

A 在 macOS 获得写入租约后，Windows 客户端只能只读。A 人工确认接管后，旧租约失效，旧设备不能再提交。

### 场景 E：Agent 拆分和完成

Agent 自动写入当前任务分析文件，但拆分子任务必须展示提案。子任务未完成时完成提案被拒绝；全部完成后用户确认，系统同步、快照、冻结并生成摘要草稿。

## 7. 发布质量门槛

- 所有领域不变量有自动化测试。
- 三层权限矩阵有 API 级契约测试。
- 任一客户端崩溃场景不会覆盖最后服务端同步版本。
- 所有 Agent 结构写入都可追溯到人工确认或明确拒绝。
- 任务图在目标规模下满足交互性能基线。
- 备份恢复演练能够恢复任务、文件快照和来源链接。
- 用户文档明确区分“本地已保存”“已同步”“已完成并快照”。

## 8. 后续评估指标

不考虑商业指标，开源自用阶段建议关注产品有效性：

- 从连接任务到获得可用上下文所需操作数。
- Agent 首次读取后正确复述目标与验收条件的比例。
- 完成任务中包含过程文档和确认摘要的比例。
- 新成员通过 Wiki 理解已完成任务所需的追问次数。
- 被权限层阻止的真实越权操作数与误拦截数。
- 工作区未同步、租约接管和恢复事件数。
- 任务图中通过搜索而非人工展开定位任务的成功率。
